# 쩌는 게임 기획서, 이렇게 쓴다 (How to Write a Great Design Document)

> **원저자:** Damion Schubert (BioWare Austin, 10년 경력 MMO 게임 디자이너)
> **출처:** GDC 2007/2008 강연 (GDC Vault)
> **번역:** 김기웅, 편집: 정진호
> **분석일:** 2026-03-23

---

## 핵심 요약

대부분의 기획서가 형편없는 이유는 **너무 복잡하고, 너무 길고, 반복적 개발을 무시하고, 변경사항이 반영되지 않기 때문**이다. 이 강연은 프로그래머가 실제로 읽고 싶어하는 기획서를 작성하기 위한 12가지 규칙과 리드를 위한 10가지 조언을 제시한다.

---

## 좋은 기획서의 5가지 목표

1. **기획에 대한 합의**를 담는다
2. **부서 간 핵심 비전 공유** 통로가 된다
3. **일정 관리**를 손쉽게 해준다
4. 개발에 **초점(focus)**을 제공한다
5. **업무 간 연관관계 및 기획 상충**을 미리 확인할 수 있게 한다

---

## 기획서 작성 12가지 규칙

### 1. 독자를 파악하라 (Know Your Target)

관심 있는 사람들과 그 이유:
| 독자 | 관심 사유 |
|:---|:---|
| 기획팀 | 기획에 대해 합의하려고 |
| 프로그래밍팀 | 게임을 구현하려고 |
| 프로듀서 | 일정/예산을 잡으려고 |
| QA | 테스트 계획을 수립하려고 |
| 외부 파트너 | 개발팀과 소통하려고 |

**핵심:** 프로그래머가 가장 중요한 독자다. 실제 게임을 구현하는 사람이기 때문. 프로그래머에게 무엇이 필요한지 물어보고, 그들이 규칙을 무시하라고 하면 따라도 된다.

---

### 2. 짧게 써라 (Keep It Short)

짧은 문서의 장점:
- 읽기 더 쉽고
- 쓰기 더 쉽고
- 관리하기 더 편하고
- 모순될 가능성이 더 적고
- 단순한 기획이 될 가능성이 더 높다

**실전 기법:**
- 군더더기(fluff) 제거
- 빈 항목 제거
- 복사-붙여넣기 금지
- 자명한 시스템에 불필요한 설명 제거

**나쁜 예시 (너무 길다):**
> 길드 초대 확인 UI. 확인 UI는 "이 길드에 가입하시겠습니까?"라고 묻고... 확인 버튼... 취소 버튼... 닫기 버튼... Esc...

**좋은 예시:**
> 길드 초대 확인 UI. 플레이어가 길드에 초대를 받으면, 확인창이 열립니다 (일반대화창.doc 참조).

**나쁜 예시 (쓸데없는 세계관 설명):**
> 아이템 제작세. 대장장이의 신, 헤파이스토스는 아테네의 장인들에게...

**좋은 예시:**
> 아이템 제작세. 아이템을 제작할 때마다, 플레이어는 해당 지역의 신전에 십일조를 납부해야 합니다.

---

### 3. 우선순위를 부여하라 (Prioritize the Design)

피처에 우선순위를 부여하고 단계별로 쪼갠다:

| 단계 | 내용 |
|:---|:---|
| **1단계 (프로토타입)** | 게임 아이디어를 검증/시연하는 데 필수 |
| **2단계 (핵심 구현)** | 콘텐츠 생성 관련 피처와 도구 |
| **3단계 (출시 필수)** | 2순위 피처에 관련된 피처들 |
| **4단계 (희망사항)** | 아마도 확장팩 |
| **5단계** | 야~ 신난다~ (꿈) |

**나쁜 예시:** 피처들을 단순 나열

**좋은 예시:** 피처들을 단계별 문서로 분리
```
기본 장비 (프로토타입)
  - 장착, 전투력 변경

고급 장비 (2단계)
  - 외형 표시, 특수 효과

장비와 길드 문장 (4단계)
  - 방패에 문장 그려넣기
```

**핵심:** 우선순위 결정은 개별 피처가 아닌 프로젝트 전체에 대한 것. 하나의 기획서 안에 여러 단계의 피처가 공존할 수 있다.

---

### 4. 보여주어라 (Illustrate)

백문이 불여일견. 활용 가능한 시각화:
- 유사 게임의 스크린샷
- Visio 다이어그램
- '예제' 텍스트

**예제 활용법:**
> 조밥은 기초 초능력과 고급 초능력을 잊어버리기로 결정했고, mindwiper를 찾아갔습니다. 기초 초능력을 삭제하려 했지만 실패 → 고급 초능력의 전제조건이었기 때문. 고급 초능력을 먼저 삭제한 후 기초 초능력 삭제 성공.

**주의:** 그림이 추상적일수록 독자가 자신의 관점을 투사하기 쉬움. UI 아티스트에게 미학적 결정권을 남겨두어야 한다.

---

### 5. 다른 사람이 어떻게 작업해야 하는지 시시콜콜 적지 말라 (Don't Tell Others How to Do Their Jobs)

**나쁜 예시:**
> 퀘스트 변수는 캐릭터 오브젝트에 있는 비트 벡터의 연결된 목록에 저장됩니다.
→ 이건 기획자가 상관할 문제가 아님!

**좋은 예시 (기획자가 제공해야 할 정보):**
> - 게임에는 약 2,500개의 퀘스트가 있습니다
> - 진행 중인 퀘스트를 한 번에 최대 20개까지 가질 수 있습니다
> - 한 퀘스트에서 최대 10번의 선택, 각 선택 결과가 퀘스트 종료까지 저장되어야 합니다
> - 완료 상태와 결과가 저장되어야 합니다

**원칙:** 기획자는 "무엇(What)"을 정의하고, "어떻게(How)"는 구현팀에 맡긴다.

---

### 6. 사용자 스토리를 활용하라 (Use User Stories)

스크럼의 INVEST 규칙:
| 약자 | 원칙 | 설명 |
|:---|:---|:---|
| **I** | Independent | 다른 사용자 스토리와 겹치지 않는 |
| **N** | Negotiable | 세부 구현보다 최종 사용자 만족이 중요 |
| **V** | Valuable | 최종 사용자의 시각에서 작성 |
| **E** | Estimatable | 프로그래머가 일정을 잡을 수 있을 정도로 상세 |
| **S** | Small | 1주일 이상 걸리지 않는 |
| **T** | Testable | 기획팀과 프로그래밍팀이 완료 여부를 확인 가능 |

**좋은 예시 (사용자 스토리 + 세부 요구사항):**
> **경험치가 해당 레벨의 한도를 넘으면, 플레이어는 레벨이 상승합니다.**
> - 플레이어는 '딩' 하는 효과음을 듣습니다
> - 플레이어는 파티클 효과를 봅니다
> - 플레이어는 능력치를 상승시킬 수 있는 속성 점수 5점을 얻습니다
> - 플레이어는 기술에 투자할 수 있는 기술 점수 3점을 얻습니다
> - 만약 10레벨에 도달하면, 특등석을 사용할 수 있게 됩니다 (특등석.doc 참조)

**핵심:** 모든 문장이 "플레이어"로 시작한다. 프로그래머가 2~5일 정도 작업할 분량.

---

### 7. 콘텐트로부터 코드를 분리하라 (Separate Code from Content)

**나쁜 예시 (공포의 글머리 기호):**
> 제련: 대장장이의 망치와 부젓가락이 필요. 고급 망치가 있으면 더 다양한 아이템 제작 가능.
> 재단: 베틀이 필요.
> 연금술: 시험관 세트 필요...
> 조각: 망치와 끌이 필요...

**좋은 예시 (규칙만 정의):**
> **제작 도구.** 어떤 제작 기술은 도구가 필요하며, 없을 경우 사용 불가 메시지가 출력됩니다.
> **고급 도구.** 어떤 제작 기술은 더 강력한 도구를 사용해서 더 강력한 아이템을 제작할 수 있습니다.

→ 구체적인 "어떤 도구가 어떤 기술에 필요한지"는 별도 데이터 테이블/부록으로 분리.

---

### 8. 좋은 서식에 시간을 투자하라 (Invest in a Good Format)

- 팀 공용 서식 사용
- 보기 좋은 서체로 변경
- 가로 구분선 사용
- 예제에는 설명선(callout) 상자 사용
- 글머리 기호 사용
- **단, 서식의 노예가 되지 말 것**

---

### 9. 명확한 용어를 사용하라 (Use Clear Terminology)

**나쁜 예시:**
> 이 주문은 높은 DPS를 갖고 있지만, 증오 감소 효과를 갖고 있어서 레이드에서 어그로를 감소시킵니다.

**원칙:**
- 평범하고 쉬운 어휘 사용
- 오덕 용어 금지
- 회사 내부 용어 삼가
- 새로운 용어는 일관적으로 사용
- 용어집(glossary)을 만들 것

---

### 10. 중복을 제거하라 (Kill Redundancy)

중복은 악이며, 혼란을 야기하고, 오류를 발생시킨다.

**원칙:** 한 문서를 '주(master)' 문서로 만들고, 다른 문서가 그 문서를 참조하게 한다.

```
[전투능력치.doc] ← 주 문서 (스탯 공식 정의)
[아이템.doc] → "자세한 것은 전투능력치.doc 참고"
```

---

### 11. 애매한 표현 금지 (No Weak Language)

**금지어:** "아마도", "가능할 수도", "그럴지도 모릅니다", 심지어 "할 수도 있습니다"

**나쁜 예시:**
> 플레이어는 이성 NPC에게 구애할 수 있을지도 모릅니다.
> 훗날에는... 추가할지도 모릅니다.

**좋은 예시:**
```
NPC와 연애하기 (프로토타입)
  플레이어는 대화를 통해 이성 NPC를 꼬실 수 있습니다.
  또한 배운 노래를 불러서 꼬실 수 있습니다.

연애 고급 (4단계)
  플레이어는 자신만의 노래를 만들 수 있습니다.
```

**핵심:** 명확한 평서문을 사용하고, '아마도'에 해당하는 피처는 후순위로 분리한다.

---

### 12. (결정의) 이유를 포함시켜라 (Capture Your Reasoning)

**단, 기획 내용과 이유를 구분해서 작성한다.**

**나쁜 예시:**
> 플레이어는 아이템을 땅에 떨어뜨릴 수 없습니다. 이렇게 하면 지저분한 것들이 눈에 안 띄고...

**좋은 예시:**
> 플레이어는 아이템을 땅에 떨어뜨릴 수 없습니다.
>
> **자주 질문하는 것들:** 왜 아이템을 땅에 떨어뜨릴 수 없나요?
> → 지저분한 것들이 눈에 띄지 않고, 수백 개의 아이템이 걸리적거리지 않게 해줍니다.

**장점:**
- 장기 프로젝트에서 "왜 이렇게 했더라?"를 기억할 수 있음
- 같은 이슈에 대한 논쟁 재발 방지

## 리드를 위한 10가지 조언

### 1. 점진적이고 반복적인 기획을 수용하라
- 바로 다음 개발 단계를 아주 상세하게 기획
- 다른 단계는 맨-먼스 수준으로 간단하게
- 먼 미래 피처에 너무 집착하지 말 것
- 기획이 바뀔 때마다 문서 재검토

### 2. 검색 가능하게 하라
기획서는 필요한 것을 찾을 수 있어야만 자료로서 가치가 있다.
- **위키(Wiki)** 추천
- 데스크탑 검색
- 기획 바이블

### 3. 가능한 자동화하라
- 정확도 향상 (특히 포스트 스크립트적으로)
- 검색 가능
- 감사(auditing)나 보고서 추가가 쉬움

### 4. 기획서는 협업의 산물이다
> "외부의 비평 없이 허공에서 쓰여진 기획서는 '적과의 접전'에서 거의 살아남지 못합니다."

### 5. 언제나 개시 회의(Kickoff Meeting)로 시작하라
3가지 필수 질문:
1. **이 시스템의 목표는 무엇입니까?** → 존재 가치를 찾고, 울타리 기둥 문제를 결정
2. **이 문서가 답변해야 하는 질문들은?** → 문서의 범위를 결정하고, 졸속 기획을 방지
3. **이 시스템이 얼마나 더 복잡해질 수 있을까요?**

복잡도의 4단계:
| 수준 | 설명 |
|:---|:---|
| 단순한 표현 | 기존에 약간의 기능 추가 |
| 경쟁력 | 1등을 약간 개선, 위험한 선택은 안 함 |
| 대안 | 크진 않지만 경쟁자와 확연히 다르게 |
| 혁신적 | 경쟁자를 밟아버리는 수준 |

### 6. 승인 과정을 거쳐라
단계별 필터링:
1. 리드 게임 디자이너 승인
2. 기획 부서 승인
3. 시니어 리드들/다른 부서 승인

### 7. 전문가와의 상담을 의무화하라
고립 상태에서 문서 작성 금지. 반드시 상담할 대상:
- 팀 내 다른 게임 디자이너
- 해당 피처를 구현할 프로그래머
- 특수 전문지식을 가진 아티스트/프로그래머
- 도구 문서라면 '사용자들'

### 8. 진척 상황을 시각화하라
포스트-잇 등 물리적/시각적 추적 방법 활용

### 9. 변경 승인 절차를 만들어라
- 게임이 반복적으로 개발되면서 기획이 바뀜
- 의사결정자들에게 변경사항을 알리는 절차 필요
- 리드 게임 디자이너가 중재자 역할

### 10. 가끔 프로세스를 재검토하라
- 모든 문서는 6개월까지만 유효 → 이후 재검토 필수
- 4~6주마다 효과 있는 것/없는 것을 점검
- 절대 잃지 말아야 할 것: **간단명료, 항상 최신, 프로그래머 친화적**

## 아이디어 발전 과정 (부록)

### 연구하라 (Research)
- 이전 게임들, **특히 실패작들**을 살펴볼 것
- 게임이 아닌 분야의 정보도 참고
- 해당 분야 전문가와 만날 것

### 브레인스토밍하라 (Brainstorm)
- 6~8인 참가
- 틀에서 벗어나서 생각
- 나쁜 아이디어란 없다
- 공감을 얻은 아이디어를 찾아낼 것

### 약점을 제거하라 (Cull the Weak)
제거 대상:
- 구현 불가능한
- 너무 야심찬
- 게임/목표와 일치하지 않거나 상충하는
- 그냥 이상한 아이디어

### 단순하게 만들어라 (Keep It Simple)
- 핵심에 집중, 가능한 빨리 플레이테스트 가능한 것을 만들 것
- 피드백을 수렴하며 반복적으로 기획하면 자연히 복잡해짐

> "만약 다른 사람들이 여러분의 기획서를 읽고 뭐라고 말할까 두려워하고 있다면, 그것은 분명 기획서가 너무 복잡하거나 너무 이상하기 때문일 겁니다."

### 반복적 개발을 수용하라 (Embrace Iteration)
> **"어떤 게임 기획도 현실과 부딪혀서 상처입지 않고 살아남을 수 없습니다."**

- 게임 기획의 어떤 부분에도 미련을 갖지 말 것
- 구현된 후에 다시 갈아엎을 준비를 할 것

## 스크럼과 문서화

스크럼은 '가벼운 문서화'를 지향하지만, 여전히 필요한 것들:
- 배급사와 외부 파트너를 위한 문서화
- QA 테스트 계획 수립을 위한 문서화
- 아이디어 산출 절차

**가장 중요한 변화:** 문서화에 사용자 스토리를 사용

## 우리 프로젝트에 적용 가능한 핵심 원칙

| 원칙 | 적용 방법 |
|:---|:---|
| **짧게 써라** | 시스템별 1,000단어 이내의 독립 문서로 분리 |
| **우선순위 부여** | Phase 1~4로 이미 구분한 개발 로드맵과 연동 |
| **사용자 스토리** | "플레이어는 ~합니다" 형식으로 모든 피처 기술 |
| **코드/콘텐트 분리** | 시스템 규칙 문서 + 데이터 테이블 분리 |
| **중복 제거** | 스탯 공식은 한 곳에만 정의, 다른 문서에서 참조 |
| **이유 포함** | FAQ 섹션으로 결정 이유를 기록 |
| **반복적 개발** | 다음 단계만 상세 기획, 나머지는 개요 수준 |
| **개시 회의** | 각 시스템 기획 전 목표/범위/복잡도 질문 |
| **위키 사용** | Reference 폴더 + 마크다운 하이퍼링크로 구현 |
| **6개월 유효기간** | 시즌 리셋(3~6개월)과 함께 기획서도 재검토 |
