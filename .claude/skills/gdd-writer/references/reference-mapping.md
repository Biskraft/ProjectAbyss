# 레퍼런스 매핑 가이드 (Reference Mapping Guide)

## 개요 (Overview)

설계 과정에서 마주치는 문제 유형별로 참조할 레퍼런스와 핵심 인사이트를 매핑한다. 문제가 발생했을 때 이 테이블을 기준으로 관련 레퍼런스를 안내한다.

---

## 1. 문제 유형별 매핑 (Problem-Reference Mapping)

| 설계 문제 | 참조 레퍼런스 | 핵심 인사이트 |
| :-------- | :------------ | :------------ |
| "이 기능이 재미있을까?" | Sakurai - Risk & Reward | 리스크 없는 보상은 재미 없다. 플레이어가 감수하는 리스크에 비례하는 보상이 있어야 의미있는 선택이 된다 |
| "복잡도가 적절한가?" | GMTK - MDA Framework | Mechanics -> Dynamics -> Aesthetics 순서로 설계한다. 메커닉이 만드는 다이내믹이 원하는 미학을 달성하는지 역추적한다 |
| "어떤 기능을 먼저?" | Tim Cain - Design Pillars | 3대 기둥(캠핑카, 총기 제작, 자원 전략) 기여도 순으로 우선순위를 정한다. 기둥에 기여하지 않는 기능은 후순위 |
| "유저가 이해할까?" | noclip 튜토리얼 사례 | 텍스트 설명 대신 체험으로 가르친다. 첫 사용 시 실패해도 안전한 환경을 제공한다 |
| "전투가 흥미로운가?" | noclip DOOM | 이동 자체가 전투이다. 정지 상태의 전투는 지루하다. 모든 이동에 전투적 의미를 부여한다 |
| "레벨 디자인?" | noclip Hitman | Core Loop(핵심 루프) + Pullout(이탈 경로) 구조. 플레이어가 목표를 달성한 후 안전하게 빠져나가는 경로 설계 |
| "AI가 무섭나?" | noclip Alien: Isolation | 다층 AI 구조로 긴장감 리듬을 만든다. AI가 항상 강하면 피로해지고, 항상 약하면 지루해진다 |
| "반복 플레이?" | noclip Hades | 내러티브와 반복 플레이를 결합한다. 매 런마다 새로운 이야기 조각을 제공하여 반복에 의미를 부여한다 |
| "오픈 월드?" | noclip Horizon | 콘텐츠 밀도가 맵 크기보다 중요하다. 넓지만 텅 빈 맵보다 좁지만 밀도 높은 맵이 낫다 |
| "밸런스 조정?" | BO3 Stats 벤치마크 | 데이터 기반 조정을 한다. 감이 아닌 수치로 판단하고, TTK/DPS 벤치마크를 기준점으로 삼는다 |
| "프로토타입?" | 개발 프로세스 사례 | 문서보다 프로토타입이 빠르다. 의심스러우면 작게 만들어서 테스트한다 |

## 2. 도메인별 빠른 참조 (Domain Quick Reference)

### 전투 도메인 (Combat)

| 참조 | 핵심 원칙 | 적용 대상 |
| :--- | :-------- | :-------- |
| noclip DOOM | 이동 = 전투, 즉각적 피드백 | 사격, 근접, 회피 시스템 |
| Sakurai - Risk & Reward | 리스크 비례 보상, 피크 모멘트 | 모든 전투 메커닉의 Risk & Reward 설계 |
| BO3 Stats 벤치마크 | TTK 기준, DPS 범위, 사거리 곡선 | 무기 밸런스, 데미지 수치 |

**전투 도메인 핵심 질문**:
- 플레이어가 이동하면서 전투하는가?
- 리스크와 보상이 비례하는가?
- 데이터로 밸런스를 검증할 수 있는가?

### 건설 도메인 (Building)

| 참조 | 핵심 원칙 | 적용 대상 |
| :--- | :-------- | :-------- |
| noclip Hitman | Core Loop + Pullout 구조 | 방어 구조물 배치, 탈출 경로 |
| Tim Cain - Design Pillars | 기둥 기여도 우선순위 | 건설 기능 우선순위 |

**건설 도메인 핵심 질문**:
- 건설한 구조물이 캠핑카 방어에 기여하는가?
- 건설 중 리스크(무방비)가 존재하는가?
- 건설 자원이 다른 용도와 경쟁하는가?

### 경제 도메인 (Economy)

| 참조 | 핵심 원칙 | 적용 대상 |
| :--- | :-------- | :-------- |
| Tim Cain - Design Pillars | 자원 전략 기둥, 트레이드오프 | 자원 순환, Faucet & Sink |
| GMTK - MDA Framework | 메커닉 -> 다이내믹 -> 미학 | 경제 시스템이 만드는 플레이어 경험 |

**경제 도메인 핵심 질문**:
- 자원 소비처가 2개 이상 경쟁하는가? (트레이드오프)
- Scrap Parts, Tech Unit, Core Module 각각의 역할이 구분되는가?
- Tech는 팀 전용으로만 사용되는가?

### UI/UX 도메인 (UI/UX)

| 참조 | 핵심 원칙 | 적용 대상 |
| :--- | :-------- | :-------- |
| GMTK - MDA Framework | 정보 계층, 플레이어 인지 부하 | HUD 레이아웃, 정보 우선순위 |
| noclip 튜토리얼 사례 | 체험 기반 학습, 점진적 복잡도 | 온보딩, 튜토리얼 |

**UI/UX 도메인 핵심 질문**:
- 전투 중 필수 정보가 HUD에서 0.5초 내에 읽히는가?
- 처음 접한 UI를 3초 내에 이해할 수 있는가?
- 색상만으로 정보를 전달하고 있지는 않은가? (접근성)

### 좀비/AI 도메인 (Zombie/AI)

| 참조 | 핵심 원칙 | 적용 대상 |
| :--- | :-------- | :-------- |
| noclip Alien: Isolation | 다층 AI, 긴장감 리듬 조절 | 좀비 AI 행동 패턴 |
| noclip DOOM | 적 유형별 역할 분담 | 좀비 유형 설계, 웨이브 구성 |

**좀비/AI 도메인 핵심 질문**:
- 좀비가 항상 같은 패턴이 아닌, 변화하는 행동을 보이는가?
- 긴장과 이완의 리듬이 존재하는가?
- 플레이어가 AI의 행동을 학습하고 대응할 수 있는가?

## 3. 레퍼런스 활용 방법 (How to Use References)

### 단계 1: 문제 식별

설계 과정에서 막히는 지점을 구체적 질문으로 변환한다.
- Bad: "전투가 재미없다"
- Good: "전투에서 플레이어의 선택지가 1개뿐이다"

### 단계 2: 매핑 테이블 참조

위 매핑 테이블에서 가장 가까운 문제 유형을 찾고, 해당 레퍼런스의 핵심 인사이트를 확인한다.

### 단계 3: 인사이트 적용

레퍼런스의 원칙을 ProjectZ 컨텍스트에 맞게 변환한다.
- 레퍼런스: "DOOM에서 이동 = 전투"
- 적용: "ProjectZ에서 자원 수집 = 전투 준비. 수집 중 좀비 위협으로 이동과 전투가 결합"

### 단계 4: 3대 기둥 검증

적용한 인사이트가 3대 기둥 중 최소 1개를 강화하는지 확인한다.
