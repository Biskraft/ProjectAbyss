> gdd-writer 스킬의 작성 규칙 확장 가이드. 5단계 구조 상세 해설과 Bad/Good 예시 모음.

# GDD 작성 규칙 확장판 (Writing Rules Extended)

## 1. 5단계 구조 상세 가이드 (Five-Stage Structure Guide)

System_*.md 문서는 반드시 5단계 구조를 따른다. 각 단계의 목적과 필수 항목을 아래에 정리한다.

---

### 1-1. 개요 (Concept)

**목적**: 이 시스템이 "왜" 존재하는지, "왜 이 방법"인지를 명확히 한다.

**필수 항목**:
- **Intent**: 시스템의 존재 이유. "왜 만드는가?"
- **Reasoning**: 선택한 방법론의 근거. "왜 이 방법인가?"
- **Cursed Problem Check**: 두 약속이 충돌하는 딜레마 식별
- **Risk & Reward**: 플레이어가 감수하는 리스크와 그에 비례하는 보상

**Bad 예시**:
```
## 개요
이 시스템은 전투를 재미있게 만들기 위해 설계되었다.
```

**Good 예시**:
```
## 개요 (Concept)

### Intent
플레이어가 제한된 탄약 상황에서 사격/회피의 우선순위를 능동적으로 판단하게 한다.

### Reasoning
무한 탄약 시스템은 "자원 기반 전략" 기둥과 충돌한다.
탄약을 유한 자원으로 설정하면 매 교전에서 "쏠 것인가, 아낄 것인가"의 의사결정이 발생한다.

### Cursed Problem
"충분한 화력 제공" vs "자원 고갈에 의한 긴장감" — 자원 고갈 긴장감을 우선하고,
화력 부족은 근접 무기 전환으로 보완한다.

### Risk & Reward
- 리스크: 탄약 소진 시 근접전 강제 전환 (생존 위협 증가)
- 보상: 정확한 사격으로 탄약 절약 시 다음 교전에서 화력 우위
```

---

### 1-2. 메커닉 (Mechanics)

**목적**: 플레이어의 행동(Verb)을 중심으로 시스템 동작을 기술한다.

**필수 항목**:
- Action: 플레이어가 하는 행동
- Reaction: 시스템이 반환하는 반응
- Effect: 게임 상태에 미치는 영향

**필수 형식**: `Player does X -> System does Y -> Effect Z`

**Bad 예시**:
```
## 메커닉
총을 쏘면 데미지를 준다. 헤드샷이면 더 세다.
```

**Good 예시**:
```
## 메커닉 (Mechanics)

### 사격 (Fire)
- Action: 플레이어가 조준 후 발사 버튼을 입력한다
- Reaction: 시스템이 히트스캔 판정을 수행하고, 적중 부위별 데미지 배율을 적용한다
- Effect: 대상의 HP가 감소하고, 적중 부위에 따른 히트마커가 HUD에 표시된다
```

---

### 1-3. 규칙 (Rules)

**목적**: 구체적 조건, 처리 로직, 시스템 간 상호작용을 명시한다.

**필수 항목**:
- 조건 -> 처리 -> 결과 형식
- 상호작용 규칙 (다른 시스템과의 연결)
- 우선순위 (충돌 시 어떤 규칙이 우선하는가)

**Bad 예시**:
```
## 규칙
- 가까이 가면 데미지가 세진다
- 멀면 약해진다
```

**Good 예시**:
```
## 규칙 (Rules)

### 거리별 데미지 감쇄
- 조건: 사격 거리가 유효 사거리(Effective_Range_m)를 초과할 때
- 처리: (사격 거리 - 유효 사거리) / (최대 사거리 - 유효 사거리) 비율만큼 데미지 감소
- 결과: 최대 사거리 이상에서 데미지 0

### 우선순위
1. 부위별 배율 적용
2. 거리 감쇄 적용
3. 방어구 감쇄 적용
```

---

### 1-4. 데이터 & 파라미터 (Parameters)

**목적**: 모든 수치를 본문과 분리하여 YAML 코드 블록에 정리한다.

**필수 항목**:
- 변수명: 영문 Snake_Case
- 단위: 주석으로 명시
- CSV 참조: 관련 SSoT 경로

**Bad 예시**:
```
기본 데미지는 45이고 헤드샷 배율은 1.5배이다. 사거리는 50m이다.
```

**Good 예시**:
```yaml
# 기본 사격 파라미터
Base_Damage: 45          # 단위: HP
Headshot_Multiplier: 1.5 # 배율
Effective_Range_m: 30    # 단위: m
Max_Range_m: 50          # 단위: m
# SSoT: ../../Sheets/Content_Stats_Weapon_RangeList.csv
```

### 1-5. 예외 처리 (Edge Cases)

**목적**: 정상 흐름 외 상황을 사전에 정의한다.

**필수 시나리오** (최소 3개):
1. 네트워크 단절/지연
2. 동시 입력/동시 이벤트
3. 자원 부족/경계값

**Bad 예시**:
```
## 예외 처리
특이한 상황이 발생하면 적절히 처리한다.
```

**Good 예시**:
```
## 예외 처리 (Edge Cases)

### 네트워크 지연 (Latency > 200ms)
- 조건: 사격 판정 시 클라이언트-서버 간 지연이 200ms 초과
- 처리: 클라이언트 사이드 히트 판정 후 서버 검증, 불일치 시 서버 우선
- 결과: 클라이언트 히트마커 표시 후 서버 롤백 시 "미적중" UI 표시

### 동시 사격 (같은 프레임 교전)
- 조건: 두 플레이어가 동일 서버 틱에 상대를 적중
- 처리: 양쪽 모두 데미지 적용 (트레이드 킬 허용)
- 결과: 양쪽 HP 동시 감소, 둘 다 사망 가능

### 탄약 0 상태에서 발사 입력
- 조건: 잔여 탄약 0인 상태에서 발사 버튼 입력
- 처리: 발사 불가, 공이치기 소리 재생, 재장전 프롬프트 표시
- 결과: 0.3초 동안 추가 발사 입력 무시 (스팸 방지)
```

## 2. 헤더 형식 규칙 (Header Format)

모든 헤더는 한글 + 괄호 영문 형식을 따른다. 최대 4단계(####)까지만 사용한다.

**Bad 예시**:
```
## Building System
### 건설
##### 세부항목   ← 5단계 금지
```

**Good 예시**:
```
## 건설 시스템 (Building System)
### 기본 건설 (Basic Building)
#### 벽 배치 (Wall Placement)
```

---

## 3. 테이블 작성 규칙 (Table Rules)

- 테이블 셀 내부에 `**Bold**` 마크다운 금지
- 숫자 열은 우측 정렬 (`:------:`)
- 텍스트 열은 좌측 정렬 (`:------`)

**Bad 예시**:
```
| **기능명** | **데미지** |
| 사격 | **45** |
```

**Good 예시**:
```
| 기능명 | 데미지 |
| :----- | :----: |
| 사격   |   45   |
```

---

## 4. 코드 블록 규칙 (Code Block Rules)

- 파라미터는 YAML 형식
- 변수명: 영문 Snake_Case (Pascal_Snake_Case)
- 주석으로 단위 명시
- CSV SSoT 경로 주석 포함

**Bad 예시**:
```yaml
데미지: 45
속도: 빠름
```

**Good 예시**:
```yaml
Base_Damage: 45        # 단위: HP
Fire_Rate_rpm: 600     # 단위: 분당 발사 수
Reload_Time_s: 2.1     # 단위: 초
# SSoT: ../../Sheets/Content_Stats_Weapon_RangeList.csv
```

---

## 5. 참조 링크 규칙 (Reference Link Rules)

- 상대 경로 사용
- CSV 참조: `../../Sheets/Content_XXX.csv`
- 문서 참조: `../Documents/System_XXX.md`
- 외부 링크 금지 (레퍼런스 문서 제외)

**Bad 예시**:
```
자세한 내용은 C:/ProjectZ/Design/Sheets/Content_Stats.csv 참조
```

**Good 예시**:
```
자세한 수치는 [Content_Stats_Weapon_RangeList.csv](../../Sheets/Content_Stats_Weapon_RangeList.csv) 참조
```

## 6. Bad/Good 예시 모음 (Example Bank)

### 예시 1: 모호한 수치
- Bad: "적절한 데미지를 준다"
- Good: "기본 데미지 45, 헤드샷 배율 1.5x"

### 예시 2: 모호한 속도
- Bad: "빠르게 재장전된다"
- Good: "재장전 시간 2.1초, 전술 재장전 1.8초"

### 예시 3: 감상문
- Bad: "이 시스템은 매우 재미있는 경험을 제공한다"
- Good: "플레이어가 3가지 선택지 중 1가지를 매 교전마다 결정해야 한다"

### 예시 4: 범위 과대
- Bad: "전투 시스템은 사격, 근접, 투척, 방어, 회피를 포함한다"
- Good: "본 문서는 사격 시스템의 히트 판정과 데미지 계산을 다룬다"

### 예시 5: 하드코딩
- Bad: "벽의 HP는 500이고 건설 시간은 3초이다"
- Good: "벽의 HP와 건설 시간은 아래 파라미터 섹션 참조"

### 예시 6: 예외 누락
- Bad: "플레이어가 발사 버튼을 누르면 총이 발사된다"
- Good: "플레이어가 발사 버튼을 누르면 총이 발사된다. 탄약 0, 재장전 중, 스턴 상태 예외는 5절 참조"

### 예시 7: 폐기 용어
- Bad: "Scrap Metal을 사용하여 제작한다"
- Good: "Scrap Parts를 사용하여 제작한다"

### 예시 8: 스마트 따옴표
- Bad: "플레이어"가 "총"을 "제작"한다
- Good: "플레이어"가 "총"을 "제작"한다

### 예시 9: 헤더 형식
- Bad: `## Combat System` (영문만)
- Good: `## 전투 시스템 (Combat System)` (한글 + 괄호 영문)

### 예시 10: Verb 누락
- Bad: "데미지 시스템은 거리에 따라 데미지가 달라진다"
- Good: "플레이어가 사격한다 -> 시스템이 거리별 감쇄를 계산한다 -> HP가 감소한다"

### 예시 11: YAML 단위 누락
- Bad: `Fire_Rate: 600`
- Good: `Fire_Rate_rpm: 600  # 단위: 분당 발사 수`

### 예시 12: CSV 링크 누락
- Bad: "기본 데미지는 파라미터 참조"
- Good: "기본 데미지는 [Content_Stats_Weapon_RangeList.csv](../../Sheets/Content_Stats_Weapon_RangeList.csv) WEP_RNG_AR_01 참조"
