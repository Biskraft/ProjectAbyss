# 설계 코칭 가이드 (Design Coaching Guide)

## 개요 (Overview)

이 가이드는 "세계 최고의 게임 디자이너" 페르소나로서 GDD 작성 시 설계 품질을 높이기 위한 코칭 프로세스를 정의한다. 모든 시스템 설계는 3단계 코칭을 거쳐 검증한다.

---

## 1. 코칭 3단계 (Three-Stage Coaching)

### 1단계: 의도 검증 (Intent Verification)

**핵심 질문**: "이 시스템이 해결하려는 저주받은 문제는?"

모든 시스템은 존재 이유가 있어야 한다. 단순히 "다른 게임에도 있으니까"는 이유가 되지 않는다.

**검증 항목**:
- 이 시스템 없이 게임이 성립하는가? -> 성립하면 시스템 존재 이유 재검토
- 어떤 플레이어 경험을 만드는가? -> "재미있다"가 아닌 구체적 경험
- 3대 기둥 중 어디에 기여하는가? -> 최소 1개와 연결 필수

**코칭 예시**:
```
"이 시스템의 Intent를 보면 '전투를 다양하게 만든다'라고 되어 있습니다.
그런데 '다양하다'는 목표가 아닌 수단입니다.
진짜 질문은: 플레이어가 매 교전에서 어떤 의사결정을 해야 하는가?"
```

---

### 2단계: Risk & Reward 검증 (Risk & Reward Verification)

**핵심 질문**: "리스크와 보상이 비례하는가?"

플레이어가 감수하는 리스크에 비례하는 보상이 없으면 시스템은 의미 없는 작업(Chore)이 된다.

**검증 항목**:
- 리스크가 구체적인가? (모호한 "위험"이 아닌 명확한 손실)
- 보상이 리스크에 비례하는가?
- 피크 모멘트(최대 리스크=최대 리턴)가 존재하는가?
- 플레이어가 리스크를 "선택"할 수 있는가?

**코칭 예시**:
```
"건설 시스템에서 벽을 세우면 적의 사격을 막을 수 있지만,
건설 중 3초간 무방비 상태가 됩니다.
리스크(3초 무방비)와 보상(엄폐물 확보)이 비례합니다.
하지만 질문: 건설 중 취소가 가능한가요?
취소 불가능하면 리스크가 '선택'이 아닌 '강제'가 됩니다."
```

---

### 3단계: 반론 검증 (Counter-Argument Verification)

**핵심 질문**: "더 단순한 방법은 없는가?"

복잡한 시스템이 항상 좋은 시스템은 아니다. 같은 경험을 더 적은 메커닉으로 달성할 수 있다면 단순한 쪽이 낫다.

**검증 항목**:
- 이 시스템에서 메커닉 1개를 빼면 경험이 무너지는가?
- 비슷한 게임에서 더 단순한 해결책이 존재하는가?
- 플레이어가 이 시스템을 이해하는 데 몇 초가 걸리는가?
- 튜토리얼 없이 첫 사용이 가능한가?

**코칭 예시**:
```
"이 제작 시스템에는 6단계 프로세스가 있습니다.
질문: 3단계로 줄이면 어떤 경험이 사라지나요?
만약 '편의성만 약간 떨어진다'라면, 3단계가 맞습니다.
만약 '설계의 깊이가 사라진다'라면, 6단계를 유지할 근거가 생깁니다."
```

## 2. 저주받은 문제 가이드 (Cursed Problems Guide)

### 정의

저주받은 문제(Cursed Problem)란 두 가지 약속이 근본적으로 충돌하는 설계 딜레마를 말한다. 정답이 없으며, 하나를 선택하면 다른 하나를 희생해야 한다.

### 핵심 원칙

1. 저주받은 문제는 "해결"이 아닌 "선택"의 대상이다
2. 선택한 쪽과 희생하는 쪽을 모두 명시적으로 문서화한다
3. 희생하는 쪽의 최소 보장선을 정의한다

### Cursed Problem ↔ 3대 기둥 매트릭스

새 시스템 설계 시 아래 매트릭스를 참조하여 해당 시스템이 어떤 Cursed Problem에 영향받는지 빠르게 판단한다.

| Cursed Problem | 기둥 1: 캠핑카 | 기둥 2: 총기 제작 | 기둥 3: 자원 전략 |
|---|:---:|:---:|:---:|
| #1 자원 풍요 vs 결핍 | 간접 | **직접** | **핵심** |
| #2 강력한 무기 vs 밸런스 | - | **핵심** | 간접 |
| #3 팀 협동 vs 솔로 능력 | 간접 | 간접 | **직접** |
| #4 캠핑카 방어 vs 공격 진출 | **핵심** | - | **직접** |
| #5 복잡한 제작 vs 접근성 | - | **핵심** | 간접 |
| #6 캠핑카 개인화 vs 전략적 효율 | **핵심** | - | **직접** |
| #7 탐색 범위 vs 기지 안전 | **핵심** | 간접 | **직접** |

- **핵심**: 이 기둥의 정체성을 정의하는 수준의 충돌
- **직접**: 이 기둥의 구현에 직접적으로 영향
- **간접**: 파생 효과로 영향
- **-**: 연결 없음

### ProjectZ 저주받은 문제 목록

#### 문제 1: 자원 풍요 vs 자원 결핍
- 약속 A: "플레이어가 원하는 무기를 자유롭게 제작한다"
- 약속 B: "한정된 자원으로 전략적 선택을 강제한다"
- 선택: 약속 B (자원 결핍)
- 희생 보장선: 기본 무기는 항상 사용 가능, 자원 0이어도 전투 참여 가능
- 기둥 연결: 기둥 3 **핵심** / 기둥 2 직접 / 기둥 1 간접

#### 문제 2: 강력한 무기 vs 밸런스
- 약속 A: "제작한 무기가 강력하다는 성취감"
- 약속 B: "모든 무기가 밸런스를 유지한다"
- 선택: 약속 A (강력한 느낌)
- 희생 보장선: TTK(Time to Kill) 범위 제한으로 극단적 불균형 방지
- 기둥 연결: 기둥 2 **핵심** / 기둥 3 간접

#### 문제 3: 팀 협동 vs 솔로 능력
- 약속 A: "팀 협동이 필수적이다"
- 약속 B: "개인 실력으로 팀을 캐리할 수 있다"
- 선택: 약속 A (팀 협동)
- 희생 보장선: Tech는 팀 전용이지만, Scrap Parts 기반 개인 장비는 자유
- 기둥 연결: 기둥 3 **직접** / 기둥 1 간접 / 기둥 2 간접

#### 문제 4: 캠핑카 방어 vs 공격 진출
- 약속 A: "캠핑카를 지키는 긴장감"
- 약속 B: "적극적 전투 참여의 재미"
- 선택: 상황별 전환 (페이즈 디자인)
- 희생 보장선: 방어 페이즈에서도 능동적 행동(함정 설치, 수리) 존재
- 기둥 연결: 기둥 1 **핵심** / 기둥 3 직접

#### 문제 5: 복잡한 제작 vs 접근성
- 약속 A: "깊이 있는 총기 설계 시스템"
- 약속 B: "누구나 쉽게 총기를 만들 수 있다"
- 선택: 약속 A (깊이)
- 희생 보장선: 프리셋 설계도 제공으로 초보자 진입 장벽 완화
- 기둥 연결: 기둥 2 **핵심** / 기둥 3 간접

#### 문제 6: 캠핑카 개인화 vs 전략적 효율
- 약속 A: "캠핑카는 '우리 팀의 집'이다 — 외형을 꾸미고 장갑을 덧대며 소속감을 부여한다"
- 약속 B: "캠핑카 업그레이드는 전략적 최적해를 따라야 한다 — 감정이 아닌 효율"
- 선택: 약속 A (애착 우선)
- 희생 보장선: 전략적 핵심 슬롯(장갑, 엔진)은 성능 기반 선택 유지, 비전투 슬롯(외형, 마킹)은 자유 개인화
- 기둥 연결: 기둥 1 **핵심** / 기둥 3 직접
- 설계 근거: Project Vision의 "또 하나의 캐릭터"라는 정의에서 비롯. 캠핑카에 감정적 투자를 한 플레이어는 방어 동기가 더 강하다. 효율만 추구하면 캠핑카는 리스폰 기계로 전락한다.

#### 문제 7: 탐색 범위 vs 기지 안전
- 약속 A: "멀리 나갈수록 희귀한 자원(Core Module 등)을 획득할 수 있다"
- 약속 B: "캠핑카에서 멀어지면 기지가 무방비 상태가 된다"
- 선택: 거리-보상 비례 곡선 (멀수록 고보상 + 고위험)
- 희생 보장선: 캠핑카 근처에도 기본 자원(Scrap Parts) 생성, 탐색 0이어도 최소한의 제작 가능
- 기둥 연결: 기둥 1 **핵심** / 기둥 3 **직접** / 기둥 2 간접
- 설계 근거: 핵심 루프 Gather → Build → Defend → Attack의 Gather 단계에서 발생하는 근본적 딜레마. "상실의 공포"를 능동적으로 체감하는 순간 — 희귀 자원을 위해 캠핑카를 떠나는 결정 자체가 리스크 선택이다.

## 3. Risk & Reward 체크리스트 (Risk & Reward Checklist)

시스템 설계 시 아래 항목을 모두 검증한다.

### 리스크 검증

- [ ] 리스크가 명확한가? (모호한 "위험"이 아닌 구체적 손실 명시)
- [ ] 리스크의 크기를 플레이어가 인지할 수 있는가? (UI/피드백 존재)
- [ ] 리스크가 플레이어의 "선택"인가? (강제 리스크는 좌절감)
- [ ] 리스크를 회피하는 대안이 존재하는가? (대안 없는 리스크는 불공정)

### 보상 검증

- [ ] 보상이 리스크에 비례하는가? (높은 리스크 = 높은 리턴)
- [ ] 보상이 즉각적인가, 지연되는가? (지연 보상은 명시적 예고 필요)
- [ ] 보상이 "느껴지는가"? (시각/청각/촉각 피드백 존재)
- [ ] 보상이 다른 시스템과 연결되는가? (고립된 보상은 의미 약화)

### 피크 모멘트 검증

- [ ] 최대 리스크 = 최대 리턴 순간이 존재하는가?
- [ ] 그 순간에 플레이어의 심박수가 올라가는가? (긴장감)
- [ ] 성공 시 "나만의 이야기"가 만들어지는가? (공유 가능한 경험)

## 4. 3대 기둥 정합성 체크 (Three Pillars Alignment Check)

모든 시스템은 3대 기둥 중 최소 1개와 연결되어야 한다. 연결되지 않으면 경고를 발생시킨다.

### 기둥 1: 캠핑카 (The Camper Van)

**질문**: 이 시스템이 캠핑카에 대한 애착 또는 공포를 강화하는가?

**연결 유형**:
- 직접 연결: 캠핑카를 건설/수리/방어하는 시스템
- 간접 연결: 캠핑카에 저장된 자원을 소비/생산하는 시스템
- 감정 연결: 캠핑카를 잃을 수 있다는 긴장감을 유발하는 시스템

**예시**:
- 건설 시스템: 직접 연결 (캠핑카 주변 방어 구조물)
- 제작 시스템: 간접 연결 (캠핑카에 저장된 Scrap Parts 소비)
- 좀비 웨이브: 감정 연결 (캠핑카가 파괴될 수 있는 위협)

### 기둥 2: 자원 기반 총기 제작 (Resource-based Gun Crafting)

**질문**: 자원 -> 설계 -> 제작 루프에 기여하는가?

**연결 유형**:
- 자원 공급: 제작에 필요한 자원을 생산하는 시스템
- 설계 확장: 새로운 제작 옵션을 열어주는 시스템
- 제작 소비: 제작된 장비를 사용하는 시스템

**예시**:
- 탐색 시스템: 자원 공급 (Scrap Parts 획득)
- 연구 시스템: 설계 확장 (새로운 무기 설계도 해금)
- 전투 시스템: 제작 소비 (제작한 무기로 전투)

### 기둥 3: 자원 기반 전략 플레이 (Resource-based Strategy)

**질문**: 한정된 자원으로 의미있는 선택을 만드는가?

**연결 유형**:
- 자원 소비 선택: A를 만들면 B를 못 만드는 트레이드오프
- 시간 투자 선택: 이 활동에 시간을 쓰면 다른 활동 불가
- 리스크 선택: 더 많은 자원을 위해 더 큰 위험 감수

**예시**:
- 제작 시스템: 자원 소비 선택 (AR vs Shotgun 선택)
- 탐색 시스템: 시간 투자 선택 (탐색 vs 방어 시간 배분)
- 위험 구역 시스템: 리스크 선택 (희귀 자원을 위한 위험 지역 진입)

### 정합성 경고

3대 기둥 중 어디에도 연결되지 않는 시스템이 발견되면:

```
[경고] 이 시스템은 3대 기둥과 연결되지 않습니다.
- 캠핑카 연결: 없음
- 총기 제작 연결: 없음
- 자원 전략 연결: 없음

조치: 시스템 존재 이유를 재검토하거나, 기둥과의 연결점을 설계하십시오.
```

## 5. 코칭 질문 뱅크 (Coaching Question Bank)

### 의도 관련 (Intent Questions)

**Q1**: "이 시스템이 없어도 게임이 성립하는가?"
- 왜 묻는가: 불필요한 시스템을 조기에 제거하기 위해

**Q2**: "플레이어가 이 시스템을 '하고 싶어서' 하는가, '해야 해서' 하는가?"
- 왜 묻는가: 강제된 시스템은 작업(Chore)이 되기 때문

**Q3**: "이 시스템이 만드는 '이야기'는 무엇인가?"
- 왜 묻는가: 기억에 남는 순간을 만들어야 시스템이 의미 있기 때문

**Q4**: "경쟁 게임에서 이 시스템을 어떻게 해결했는가?"
- 왜 묻는가: 바퀴를 재발명하지 않기 위해

**Q5**: "이 시스템의 핵심 동사(Verb) 하나만 고르면?"
- 왜 묻는가: 핵심 행동이 불명확하면 시스템 정체성이 흔들리기 때문

### Risk & Reward 관련 (Risk & Reward Questions)

**Q6**: "이 시스템에서 가장 위험한 순간은 언제인가?"
- 왜 묻는가: 피크 모멘트를 식별하기 위해

**Q7**: "실패했을 때 플레이어가 잃는 것은 구체적으로 무엇인가?"
- 왜 묻는가: 모호한 리스크는 긴장감을 만들지 못하기 때문

**Q8**: "성공했을 때 '소리 지를 만큼' 기쁜가?"
- 왜 묻는가: 보상의 감정적 강도를 검증하기 위해

**Q9**: "플레이어가 리스크를 '선택'하는가, '당하는가'?"
- 왜 묻는가: 강제 리스크는 좌절감을 유발하기 때문

**Q10**: "리스크 없이 보상만 받는 경로가 존재하는가?"
- 왜 묻는가: 무위험 보상은 시스템을 무의미하게 만들기 때문

### 밸런스 관련 (Balance Questions)

**Q11**: "이 시스템의 최강 전략이 뻔한가?"
- 왜 묻는가: 최적해가 명확하면 선택의 여지가 사라지기 때문

**Q12**: "숫자를 2배로 올리면 어떤 일이 벌어지는가?"
- 왜 묻는가: 극단값 테스트로 시스템 견고성을 검증하기 위해

**Q13**: "초보자와 고수의 성과 차이가 느껴지는가?"
- 왜 묻는가: 스킬 표현 공간이 있어야 숙련도에 의미가 생기기 때문

**Q14**: "이 시스템이 다른 시스템을 무력화하지는 않는가?"
- 왜 묻는가: 한 시스템이 모든 것을 해결하면 다른 시스템이 죽기 때문

**Q15**: "1000판 후에도 이 시스템이 흥미로운가?"
- 왜 묻는가: 장기 리텐션에서의 깊이를 검증하기 위해

### UX 관련 (UX Questions)

**Q16**: "튜토리얼 없이 이 시스템을 처음 사용할 수 있는가?"
- 왜 묻는가: 직관성이 낮으면 진입 장벽이 높아지기 때문

**Q17**: "이 시스템의 피드백이 0.5초 안에 전달되는가?"
- 왜 묻는가: 지연된 피드백은 인과관계 인식을 방해하기 때문

**Q18**: "색맹 플레이어도 이 시스템을 사용할 수 있는가?"
- 왜 묻는가: 접근성은 선택이 아닌 필수이기 때문

**Q19**: "이 시스템의 상태를 HUD에서 즉시 확인할 수 있는가?"
- 왜 묻는가: 보이지 않는 정보는 전략적 판단에 사용할 수 없기 때문

**Q20**: "조작이 3버튼 이내로 완료되는가?"
- 왜 묻는가: 조작 복잡도가 높으면 전투 중 사용이 불가능하기 때문

### 3대 기둥 전용 (Pillar-Specific Questions)

**Q21**: "이 시스템이 캠핑카를 잃을 때의 공포를 강화하는가, 약화하는가?"
- 왜 묻는가: 캠핑카의 핵심 감정은 "상실의 공포"다. 시스템이 이 공포를 희석시키면 (예: 캠핑카 파괴 후에도 리스폰 대안 존재) 기둥 1의 정체성이 무너진다
- 기둥: 캠핑카

**Q22**: "이 시스템이 캠핑카에 대한 감정적 투자를 유도하는가?"
- 왜 묻는가: "또 하나의 캐릭터"가 되려면 플레이어가 캠핑카에 시간/자원/선택을 투자해야 한다. 투자 없는 애착은 성립하지 않는다
- 기둥: 캠핑카

**Q23**: "이 시스템에서 플레이어는 '총을 줍는' 경험인가, '총을 설계하는' 경험인가?"
- 왜 묻는가: Project Vision은 "수동적 루팅"이 아닌 "능동적 공학 판타지"를 명시한다. 완제품을 드롭하는 시스템은 기둥 2를 위반한다
- 기둥: 총기 제작

**Q24**: "이 시스템이 총기의 '구조' 변경에 기여하는가, 단순 '수치' 변경에 그치는가?"
- 왜 묻는가: 스코프를 다는 커스터마이징과 배럴/리시버를 교체하는 구조 변경은 다르다. 기둥 2는 후자를 요구한다
- 기둥: 총기 제작

**Q25**: "이 시스템에서 자원을 소비할 때, 플레이어가 '무엇을 포기하는지' 인지하는가?"
- 왜 묻는가: 기회비용이 체감되지 않으면 전략적 선택이 아닌 단순 소비가 된다. "이 Scrap Parts로 AR을 만들면 방벽을 못 세운다"는 긴장감이 핵심이다
- 기둥: 자원 전략

**Q26**: "이 시스템이 Gather-Build-Defend-Attack 루프 중 어디에 위치하며, 인접 단계와 자원이 순환하는가?"
- 왜 묻는가: 핵심 루프에서 고립된 시스템은 자원이 정체되거나 새어나간다. 모든 시스템은 루프의 한 지점에 기여하고, 자원을 다음 단계로 넘겨야 한다
- 기둥: 자원 전략
