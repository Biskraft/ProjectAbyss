|:---|:---|:---|:---|

**웹 게임 특수 사항:** PC 브라우저에는 TV 오버스캔이 없으므로 Action Safe가 사실상 100%. 그러나 다음 이유로 마진 유지를 권장:
- 브라우저 창 크기 변동
- 향후 콘솔/모바일 포팅
- 화면 가장자리 UI는 주변 시야 한계로 시인성 낮음

**ECHORIS 권장:** 모든 HUD 요소를 각 변에서 최소 **8-16px** (640x360 기준) 안쪽에 배치.

|:---|:---|:---|:---|
| **콘솔** | 1080p | **26px** | Xbox XAG 101 |

| 속성 | 최소 기준 |
|:---|:---|

**ECHORIS 적용 (640x360 기준):**
- HUD 본문 텍스트: 최소 **6px** (x3 = 18px @1080p, PC 최소 충족)
- 메뉴/제목 텍스트: 최소 **9px** (x3 = 27px @1080p, 콘솔 기준 충족)
- 현재 BitmapText(Press Start 2P) 사용 중 - GPU 직접 렌더링으로 성능 적정

| 표준 | 최소 크기 | 비고 |
|:---|:---|:---|
| WCAG 2.1 AAA | 44x44px (CSS) | 웹 접근성 최고 등급 |
| Google Material | 48x48dp (~9mm) | 손가락 끝 크기 |
| WCAG 2.2 AA | 24x24px | 최소 요구 |
| 요소 간 간격 | 8px 이상 | 오터치 방지 |

**ECHORIS:** 현재 `SLOT_SIZE = 20px` (베이스). 1080p 3x = 60px → WCAG AAA 충족. 720p 2x = 40px → AAA 미달(44px), AA 충족(24px).

### A3. 화면 점유율 (Screen Real Estate)

50개 인기 게임 25년치 분석 결과 (Josh Moskowitz 연구):

| 장르 | HUD 점유율 |
|:---|:---|
| **전체 평균** | **6%** |
| 플랫포머/액션 (HK, DC, Celeste) | **1-7%** |
| FPS | 3-8% |
| RTS/전략 | 15-25% |
| RE7/Tomb Raider | 0% (영구 HUD 없음) |

**핵심 원칙:**
1. 영구(persistent) HUD는 **5-10% 이하**가 이상적
2. 동적(dynamic) HUD 활용 - 필요할 때만 표시
3. 디에제틱(diegetic) 표시 - 게임 세계 안에 통합 (Dead Space 등 체력바)

**ECHORIS 면적 예산 (640x360 = 230,400px 총면적):**

| 목표 | 픽셀 수 | 실질 의미 |
|:---|:---|:---|
| 3% (미니멀) | ~6,912px | HP바 + 골드 정도 |
| 6% (평균) | ~13,824px | HP + MP + 미니맵 + 골드 |
| 10% (최대) | ~23,040px | 풀 HUD |

**출처:** "Size Does Matter: Analysis of Videogame HUDs" (Josh Moskowitz)

### A4. 시선 추적 (Eye Tracking)

#### 2D 횡스크롤러 시선 분포 모델

```
┌──────────────────────────────────────────┐
│  [2차]                    [3차]          │  ← 거의 안 봄 (1-3%)
│                                          │
│                                          │
│            ★★★★★★★                     │
│         ★★ 캐릭터 주변 ★★               │  ← 집중 (70-80%)
│            ★★★★★★★                     │
│                                          │
│  [1차]                    [2차]          │  ← 가끔 확인 (15-20%)
└──────────────────────────────────────────┘
```

#### 핵심 발견

- 플레이어 시선은 **캐릭터 주변에 고정** (화면 중앙-하단)
- 화면 상단 구석은 **가장 적게 응시하는 영역**
- 주변 시야(peripheral vision)에서는 **색상보다 명도 대비**가 더 효과적
- **움직임/깜빡임**은 주변 영역에서도 즉시 포착 (생존 본능)
- 한 프로덕션에서 좌상단에 배치한 스텔스 인디케이터를 **테스트 플레이어 중 아무도 보지 못함**

**교훈:** 중요 정보를 화면 구석에 넣으면 안 됨. 넣더라도 **플래시/애니메이션으로 보강** 필수.

**출처:** Player Research, Game Developer(Gamasutra) Eye Tracking 기사

### A5. 피츠의 법칙 (Fitts's Law)

```
MT = a + b * log2(D/W + 1)
```
- MT: 이동 시간, D: 거리, W: 타겟 너비

**게임 UI 적용:**

| 원칙 | 적용 |
|:---|:---|
| 화면 가장자리/구석 = 무한 크기 타겟 | 닫기 버튼은 구석에 배치 → 빠른 접근 |
| 자주 쓰는 버튼은 크게 | 메인 액션 버튼 = 다른 버튼의 **1.5배** 이상 |
| 관련 요소는 가까이 | 연속 상호작용 요소끼리 거리 최소화 |
| 팝업은 커서 근처 | 인벤토리 열릴 때 마우스 위치 근처에 표시 |

**2D 횡스크롤러 특이사항:** 실시간 전투 중에는 HUD를 클릭하지 않으므로 피츠의 법칙은 **메뉴/인벤토리**에 주로 적용. 인게임 HUD는 **가독성(readability)**이 더 중요.

**출처:** Paul Fitts (1954), Nielsen Norman Group, Interaction Design Foundation

### A6. 정보 계층 구조 (Information Hierarchy)

| 단계 | 내용 | 배치 | 인식 방식 |
|:---|:---|:---|:---|
| **1차** | HP, 현재 무기, 위험 경고 | 캐릭터 근처 또는 화면 하단-좌측 | 주변 시야 상시 인식 |
| **2차** | MP, 골드, 미니맵, EXP | 화면 구석 (상단 좌/우) | 필요시 시선 이동 |
| **3차** | 퀘스트, 알림, 버프 타이머 | 화면 가장자리 또는 팝업 | 이벤트 시에만 인식 |

**좌측 > 우측:** 좌측이 더 자주 응시됨 → 가장 중요한 정보는 좌측
**상단 > 하단:** 일반적이나 2D 횡스크롤러에서 캐릭터가 하단이면 하단 시인성 상승

**출처:** Polydin Game HUD Design Guide, MY.GAMES HUD 분석

### A10. 게슈탈트 원칙 (Gestalt)

| 원칙 | 게임 UI 적용 |
|:---|:---|
| **근접(Proximity)** | HP/MP를 함께 묶어 배치. 미니맵과 나침반 인접 |
| **유사(Similarity)** | 같은 크기/색상의 인벤토리 슬롯 = 하나의 그리드 |
| **연속(Continuity)** | HP바의 좌→우 감소 방향 |
| **폐쇄(Closure)** | 원형 쿨다운 게이지 |
| **공통 영역** | 패널/박스로 관련 정보 묶기 |

**간격 규칙:**
- 그룹 내 간격: 요소 크기의 **25-50%** 이하
- 그룹 간 간격: 그룹 내 간격의 **2배 이상**

**ECHORIS 현재:** `SLOT_SIZE=20, SLOT_GAP=2, PADDING=8` → 슬롯 간 2px(10%)는 그룹핑 효과적. 패딩 8px는 외부 분리 적절.

**출처:** Nielsen Norman Group, Interaction Design Foundation

### A11. 애니메이션 타이밍

| 유형 | 권장 시간 | 비고 |
|:---|:---|:---|
| 즉각 피드백 (토글) | **100ms** | 직접 조작 느낌 하한 |
| 진입 애니메이션 | **200-300ms** | 부드럽지만 지연감 없음 |
| 퇴장 애니메이션 | **150-250ms** | 진입보다 빠르게 |
| HUD 값 변화 (HP 감소) | **150-300ms** | 주변 시야 인식 가능 |
| 최대 허용 | **400ms** | 이상은 "느리다" 체감 |
| 금지 | **500ms+** | 짜증 유발 |
| 깜빡임 제한 | 초당 **3회 이하** | 광과민성 발작 방지 (WCAG 2.3.1) |

**이징(Easing):**
- 진입: **ease-out** (빠르게 시작 → 천천히 멈춤)
- 퇴장: **ease-in** (천천히 → 빠르게 사라짐)
- 강조: **ease-in-out** (부드러운 맥동)

### C1. 브라우저 크롬 / 뷰포트 손실

| 브라우저 | 탭+주소창 | 총 손실 (1080p) | 가용 높이 |
|:---|:---|:---|:---|
| Chrome | 72-120px | 960-1008px | **2x 정수(720px) 가능** |
| Firefox | 81-101px | 979-999px | 2x 가능 |
| Safari | 82-106px | 974-998px | 2x 가능 |
| Edge | 88-116px | 964-992px | 2x 가능 |

**1080p 브라우저 창에서 640x360의 정수 배율:**
- 윈도우 모드: 높이 ~960px → 최대 **2x** (720px). 3x(1080px) 불가
- **전체화면 모드: 1920x1080 → 정확히 3x** (딱 맞음)

→ 전체화면 진입이 UX 품질 결정적 분기점.

### C2. 정수 배율 강제 (현재 코드 문제점)

현재 `Game.ts`의 `handleResize()`:
```typescript
scale = Math.min(w / 640, h / 360)  // 비정수 허용됨
```

**문제:** scale이 2.67x, 1.83x 등 비정수가 되면 픽셀아트에 서브픽셀 블리딩 발생.

**권장 수정:**
```typescript
const rawScale = Math.min(w / 640, h / 360);
const scale = Math.max(1, Math.floor(rawScale));  // 정수만 허용
```

나머지 영역은 레터박스(블랙바) 처리.

### C3. 전체화면 API

| 타이밍 | 권장도 | 이유 |
|:---|:---|:---|
| 타이틀 화면 버튼 | 최우선 | 사용자 제스처에서만 호출 가능 |
| 첫 클릭 시 자동 | 수용 가능 | 단 1회만 |
| 게임 시작 시 자동 | 비권장 | 브라우저가 차단 |

**ESC 충돌:** 전체화면에서 ESC = 전체화면 해제. 게임 메뉴 키와 충돌.
→ `fullscreenchange` 이벤트 감지 + 전체화면 해제 시 메뉴 열지 않도록 처리.

**전체화면 시 정수 배율 (주요 해상도):**

| 해상도 | 640x360 배율 | 결과 |
|:---|:---|:---|
| 1920x1080 | **3x** | 완벽 |
| 2560x1440 | **4x** | 완벽 |
| 3840x2160 | **6x** | 완벽 |

### C4. 브라우저별 픽셀아트 렌더링

ECHORIS의 현재 CSS:
```css
image-rendering: pixelated;    /* Chrome, Edge, Firefox */
image-rendering: crisp-edges;  /* Safari 폴백 */
```

| 브라우저 | `pixelated` | `antialias: false` | WebGL 2.0 |
|:---|:---|:---|:---|
| Chrome 115+ | 완벽 | 정상 | 완전 지원 |
| Firefox 93+ | 완벽 | 정상 | 완전 지원 |
| Safari 15+ | 완벽 | **가끔 무시됨** | Safari 15+ |
| Edge (Chromium) | Chrome 동일 | 정상 | 완전 지원 |

**Safari 주의:** `antialias: false`가 무시되는 버그 보고. `scaleMode: 'nearest'` + CSS `pixelated`로 이중 보호 필요 → ECHORIS는 이미 양쪽 적용 중.

### C6. HUD 성능 (PixiJS WebGL)
