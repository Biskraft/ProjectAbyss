## 1. 소셜 허브 레퍼런스 분석

### 1.1 워프레임 — 오비터(Orbiter) + 릴레이(Relay): 개인 공간과 공용 공간의 이중 구조

워프레임은 허브 공간을 **개인 공간(오비터)**과 **공용 공간(릴레이)**으로 완전히 분리한 가장 명확한 사례다.

#### 오비터 (Orbiter) — 개인 기지

- **성격:** 플레이어 전용 우주선. 완전히 사적인 공간
- **기능:** 장비 관리, 제작(파운드리), 코덱스, 미션 네비게이션, 동료(동반자) 보관
- **커스터마이징:** 수천 종의 장식품으로 내부 인테리어 자유 배치. 비품, 가구, NPC 배치 가능
- **사회성:** 초대받은 플레이어만 입장 가능. "내 공간에 누구를 들일지" 플레이어가 통제

#### 릴레이 (Relay) — 공용 사회 공간

- **성격:** 수십~수백 명의 플레이어가 동시에 접속하는 공용 허브
- **구역 구성:**
  - **격납고(Hangar):** 진입 지점. 착륙정 도킹 연출로 "도착했다"는 감각 제공
  - **콩코스(Concourse):** 중앙 광장. NPC와 플레이어가 뒤섞이는 핵심 사교 공간. 거대 Rhino 조각상이 랜드마크 역할
  - **벤더 날개(Vendor Wings):** 신디케이트별 전용 구역으로 기능 분산
  - **미션 컨트롤:** 릴레이 내에서도 네비게이션 콘솔 접근 가능
- **사회적 기능:** 제스처/이모트, 채팅, 트레이딩, 바로(Baro Ki'Teer) 한정 벤더 방문

#### 설계 교훈

| 구분 | 오비터 | 릴레이 |
|:-----|:------|:------|
| 목적 | 준비/관리/표현 | 교류/거래/이벤트 |
| 인구 | 혼자 또는 초대된 소수 | 수십~수백 명 무제한 |
| 커스터마이징 | 매우 높음 | 불가 (공용 공간) |
| 사회적 압력 | 없음 (안전한 공간) | 높음 (타인에게 보임) |

**핵심 통찰:** 워프레임 커뮤니티에서 "허브들이 게임을 파편화한다"는 비판이 반복적으로 제기된다. 오비터, 도조(길드 기지), 릴레이, 오픈 월드 허브(세투스, 포르투나)가 각각 분리되어 있어 어디로 가야 하는지 혼란스럽다는 것이다. **허브 기능이 분산될수록 플레이어의 인지 부하가 증가한다는 중요한 역설적 교훈을 제공한다.**

### 1.2 몬스터헌터 — 집회소(Gathering Hall): 퀘스트 준비와 소셜 기능의 통합

몬스터헌터 시리즈는 허브 설계 철학이 시리즈 전반에 걸쳐 가장 크게 진화한 레퍼런스다.

#### 구버전 시리즈 (MH Tri ~ MH4U)

- **마을(Village):** 솔로 전용. 마을 퀘스트(Village Quests)만 진행 가능
- **집회소(Gathering Hall):** 멀티플레이 전용 별도 공간. 마을과 집회소 퀘스트가 완전히 분리된 병렬 진행
- **문제점:** 솔로 진행자가 멀티플레이 콘텐츠에 자연스럽게 연결되지 않음. 전환 시 로딩 화면 필요

#### MHW(World) — 통합 허브 설계

- **아시나 거점(Astera):** 마을과 집회소 기능을 단일 공간에 통합
- **집회소(The Gathering Hub):** 최대 16인이 같은 세션에 존재. 온천, 족욕탕, 사우나 등 사교 공간 내장
- **아이스본(Iceborne) — 셀리아나(Seliana):** 추가 확장팩 허브. "추위 속의 따뜻한 안식처" 콘셉트
  - 대장장이, 자원 센터, 식물원, 아르고시(상점선) 등 주요 시설을 한 공간에 집약
  - **로딩 화면 수 대폭 감소** — UX 개선의 핵심 지표로 플레이어들이 명시적으로 언급

#### 집회소 구조 원칙

- 불을 중심으로 하는 원형 레이아웃 → 자연스러운 집결 지점 형성
- 요리사(Meowscular Chef)가 집회소 중앙에 위치 → 전투 전 버프가 소셜 공간에서 이루어지는 구조
- 제스처(Gestures)와 단체 행동으로 비언어적 소통 지원

#### 설계 교훈

**"준비하러 왔다가 자연스럽게 사교하게 되는" 동선 설계.** 플레이어가 퀘스트를 위해 반드시 거쳐야 하는 곳에 사교 기능을 녹여 넣어 허브 방문 빈도를 높인다. 로딩 화면 감소 = 허브 진입 마찰 감소 = 체류 시간 증가의 직접적 연결 고리.

### 1.3 PSO2 로비 — 대규모 플레이어 가시성 + 상점 밀집 구조

판타지 스타 온라인 2(PSO2)는 허브 공간에서 **"많은 플레이어가 보이는 것 자체가 콘텐츠"**라는 설계 철학을 가장 적극적으로 구현한 사례다.

#### 로비 구조

- **2층 구조:** 게이트 구역(퀘스트 수락, 미션 출발)과 쇼핑 구역(각종 상점)으로 분리. 엘리베이터로 연결
- **블록(Block) 시스템:** 여러 블록이 병렬로 존재. 각 블록은 이름으로 용도 구분 (거래 블록, 사교 블록 등)
- **플레이어 숍(My Shop):** 플레이어가 직접 NPC 카운터를 통해 상점 개설. 로비 내에 쇼핑 구역이 형성

#### 특징적 UX 패턴

- **로비 액션(Lobby Actions):** 수백 종의 이모트/포즈. 로비에서 아이들 중인 플레이어들이 자신의 개성을 표현
- **시각적 밀도:** 많은 플레이어가 각자의 코스메틱을 갖추고 대기 중인 로비는 그 자체로 "이 게임이 활발하다"는 신호. 신규 플레이어에게 강렬한 첫인상 제공
- **상점 밀집 효과:** 쇼핑 구역에 플레이어 숍이 집중되면 거래 효율 향상 + 자연스러운 경제 허브 형성
- **코스메틱 전시 공간:** 로비 = 코스메틱 쇼케이스. 다른 플레이어의 외형을 보고 구매 욕구가 자극되는 설계

#### 설계 교훈

**플레이어의 존재 자체를 콘텐츠로 만드는 설계.** 그러나 동시에, 너무 많은 플레이어가 한 공간에 있으면 성능 문제와 시각적 혼잡이 발생한다. PSO2는 블록 분리로 이를 관리했지만, 이는 동시에 커뮤니티 분산 문제를 야기했다.

### 1.4 데스티니 2 타워 — 비전투 안식처 + NPC 기능 분산

데스티니 2의 타워(The Tower)는 "전투가 없는 안전한 공간"으로서의 허브 정체성을 가장 명확하게 구현한 사례다.

#### 구역 구성

| 구역 | 주요 NPC | 기능 |
|:-----|:--------|:----|
| 플라자(Plaza) | 자발라(Zavala), 반시-44(Banshee-44) | 메인 벤더, 무기 거래 |
| 바자르(Bazaar) | 이코라 레이(Ikora Rey), 에리스 몬(Eris Morn) | 마법/특수 벤더 |
| 격납고(Hangar) | 케이드-6(Cayde-6, 과거), 아만다 홀리데이 | 정찰/익스트라 |

#### 디자인 철학

- **비전투 안전지대:** 총을 쏘거나 적이 없는 순수한 안식처. "여기서는 쉬어도 된다"는 심리적 신호
- **비공식 미니게임:** 바닥 용암(Floor is Lava), 축구, 아이스하키 등 플레이어 자발적 놀이 문화 형성. 개발팀이 의도한 것이 아닌 플레이어들이 만들어낸 문화
- **NPC 이야기 전달 공간:** 벤더 NPC들이 단순히 거래만 하는 것이 아닌, 스토리 대사와 개성을 통해 세계관을 전달

#### 알려진 UX 문제

- **NPC 간 거리 문제:** 커뮤니티에서 지속적으로 제기된 불만. 플라자에서 바자르, 격납고까지의 동선이 길어 자주 사용하는 벤더에 빠르게 접근하기 어렵다는 지적
- **신규 사회 공간 논의:** 데스티니 2 개발팀이 새로운 사회 공간을 검토했을 때, 기존 타워의 벤더 접근성 문제가 핵심 개선 과제로 논의됨

#### 설계 교훈

**공간의 규모 != 사용성.** 타워는 시각적으로 인상적이지만, 플레이어가 자주 방문하는 벤더가 분산되어 있어 실용적 마찰이 크다. **가장 자주 사용하는 기능은 가장 가까운 곳에.** 이 원칙의 중요성을 역설적으로 증명하는 사례.

### 1.5 FFXIV 주거지 — 플레이어 소유 공간 + 커뮤니티 형성

파이널 판타지 XIV(FFXIV)의 하우징 시스템은 플레이어 소유 공간이 어떻게 커뮤니티의 핵이 되는지를 가장 심층적으로 보여주는 레퍼런스다.

#### 주거지 구조

- **개인 주택:** 플레이어가 자신만의 집을 구입하여 자유롭게 꾸밈. 파티션을 초대하거나 공개 개방 가능
- **자유 회사(Free Company) 집:** 길드 전용 공간. 회원들이 공동으로 사용하는 거점. 특별 시설(제작소, 창고 등) 설치 가능
- **아파트먼트:** 플롯(토지) 구입 없이 이용 가능한 소형 개인 공간. 진입 장벽 낮음

#### 커뮤니티 형성 패턴

- **이벤트 공간:** 플레이어들이 자발적으로 집을 코디 컨테스트 행사장, 롤플레이 공간, 카페 콘셉트 인테리어로 꾸며 다른 플레이어를 초대
- **패치 7.1의 설계 방향:** 기둥 없는 미니멀 인테리어 스타일 추가 → 대형 집회 공간을 원활하게 연출할 수 있도록 플레이어 표현 자유도 향상
- **자유 회사 하우징의 사회적 기능:** "가입하면 우리 집이 생긴다"는 길드 가입 동기 제공

#### 하우징 시스템의 문제점

- **토지 부족:** 전 세계적으로 주택 플롯이 부족하여 구매 경쟁이 치열. "집을 살 수 없다"는 불만이 FFXIV 최대 UX 문제 중 하나
- **해결 방향:** 아파트먼트 확장, 내부 공간만 제공하는 개인 공간 시스템 검토

#### 설계 교훈

**플레이어 소유 공간은 게임에 대한 심리적 귀속감을 폭발적으로 높인다.** 그러나 소유 가능한 공간의 희소성을 잘못 관리하면 좌절감의 원천이 된다. **접근 장벽 없는 최소한의 개인 공간 보장이 선행되어야 한다.**

### 1.6 Path of Exile 은신처 — 개인 커스텀 공간 + 거래 기능

PoE의 은신처(Hideout) 시스템은 개인 공간이 어떻게 기능성과 미학을 동시에 충족시키는지를 보여주는 핵심 레퍼런스다.

#### 은신처 기능

- **기능 배치 자유도:** NPC 벤더, 스태시(창고), 지도 장치, 제작대를 플레이어가 원하는 위치에 자유 배치
- **효율 최적화:** 고숙련 플레이어들은 모든 핵심 시설을 최소 이동 거리에 배치하여 파밍 효율을 극대화. "은신처 레이아웃 공유" 문화가 자연 발생
- **거래 허브:** 거래 시 판매자의 은신처를 방문하는 것이 PoE의 기본 거래 관행. "격차(랙) 없는 개인 공간"이기 때문에 대형 마을 입장 대비 성능이 안정적
- **최대 장식품 수:** 750개 (은신처당)
- **은신처 공유/경쟁:** 은신처 디자인 대회, 공유 사이트(Hideoutshowcase.com) 등 독립적 부문화 형성

#### PoE2에서의 변화

PoE2에서도 은신처 시스템이 핵심 UX로 유지. 은신처 해금 방식과 데코 시스템 개선 예정.

#### 설계 교훈

**개인 공간의 기능 배치 자유도가 "나만의 최적화된 기지"라는 장인 정신을 자극한다.** 특히 은신처 배치 최적화가 제2의 미니게임이 되는 현상은 UX 설계가 의도치 않게 메타게임을 만들어낸 흥미로운 사례다.

## 2. 허브 핵심 기능 설계

### 2.1 기능 우선순위 분류

허브에서 제공해야 할 기능을 방문 빈도와 중요도에 따라 3계층으로 분류한다.

```
[1계층 — 매 세션 필수 접근]
   이노센트 관리 / 장비 강화
   아이템계 진입 (파티 구성 + 아이템 선택)
   창고/인벤토리 관리

[2계층 — 정기적 접근]
   거래소 / 직접 거래
   NPC 상점 (소모품 보충)
   파티 모집 게시판
   이노센트 팜 현황 확인

[3계층 — 산발적 접근]
   외형 커스터마이징 / 코스메틱
   길드/클랜 관리
   업적/퀘스트 확인
   사교 공간 (이모트, 잡담)
```

**UX 원칙:** 1계층 기능은 진입 지점에서 3초 이내 접근 가능해야 한다. 2계층은 한 번의 이동으로, 3계층은 자연스럽게 탐색되는 위치에 배치한다.

---

### 2.2 장비 관리 / 강화 (이노센트 관리 + 아이템 합성)

#### 이노센트 관리소

아이템계에서 획득한 이노센트를 관리하는 핵심 시설. 다음 기능을 단일 UI에서 제공해야 한다:

| 기능 | 설명 | UX 고려사항 |
|:-----|:----|:-----------|
| 이노센트 인벤토리 | 보유 이노센트 목록, 정렬/필터 | 타입별 색상 코딩, 스탯 수치 명시 |
| 이노센트 이식 | 장비 슬롯에 이노센트 배치 | 드래그 앤 드롭 + 키보드 대안 제공 |
| 이노센트 이전 | A 장비 → B 장비로 이동 | 비용(HL) 표시 후 확인 절차 |
| 이식 시뮬레이터 | 이식 결과 스탯 미리보기 | 변화량 시각화 (+X.X% 강조) |
| 이노센트 팜 현황 | 팜 시설 수확 대기 이노센트 수 | 배지 숫자로 상시 노출 |

디스가이아 5에서 이노센트 팜이 추가된 이유: 이노센트를 여러 메뉴에서 왔다 갔다 관리하는 마찰이 컸기 때문. Project Abyss에서는 이노센트 관련 모든 작업을 **단일 허브 시설에서 원스톱으로** 처리해야 한다.

#### 장비 합성 / 분해

- 합성: 2개 이상의 장비를 소재로 사용하여 새 장비 제작. 재료 장비의 이노센트/스탯 일부 계승 여부 표시 필수
- 분해: 불필요한 장비를 HL 또는 강화 소재로 전환. "이노센트를 먼저 빼겠습니까?" 확인 팝업 필수 (이노센트 유실 방지)

## 3. 파티 형성 & 매치메이킹

### 3.1 수동 파티 모집 게시판 vs 자동 매칭

#### 두 모델의 비교

| 항목 | 수동 게시판 | 자동 매칭 |
|:-----|:---------|:--------|
| 사회적 상호작용 | 높음 (모집 글, 채팅, 선택) | 없음 (알고리즘이 처리) |
| 매칭 속도 | 느림 (모집자/응모자 대기) | 빠름 |
| 파티 구성 통제권 | 높음 (모집자가 선택) | 낮음 |
| 커뮤니티 형성 | 강함 (얼굴 익히기, 단골 파티) | 약함 (일회성 만남) |
| 비핵심 시간대 | 인원 부족 시 장시간 대기 | 알고리즘이 풀 관리 |
| Bartle 사교가 | 충족 | 미충족 |

#### SDT(자기결정이론) 관점

자동 매칭은 유능감(빠른 파밍)을 지원하지만 자율성(파티원 선택)과 연결감(관계 형성)을 해친다. GDC에서 제시된 "친구 관계는 반복 접촉을 통해 형성된다"는 원칙상, 매번 다른 사람과 매칭되는 시스템은 장기 커뮤니티 형성을 방해한다.

#### 권장 모델: 하이브리드 (수동 우선 + 자동 보완)

```
[파티 구성 우선순위]

1순위: 친구/길드 멤버 직접 초대
2순위: 파티 게시판 수동 모집
3순위: 빠른 매칭 (자동 알고리즘, 대기 2분 이상 시 플레이어가 선택)

→ "자동 매칭으로 합류하겠습니까?" 확인 팝업
→ 거부하면 계속 수동 대기
```

이 구조는 커뮤니티 형성을 위해 수동 상호작용을 기본값으로 하면서, 동시 접속자가 적은 시간대에도 파티를 구성할 수 있는 안전망을 제공한다.

---

### 3.2 아이템계 진입 시 파티 구성 UX 플로우

#### 전체 플로우

```
[허브에서 파티 구성] → [아이템 선택] → [아이템계 진입] → [세션 내 플레이]

상세:

Step 1: 파티 게시판 접근 또는 친구 초대
  ↓
Step 2: 파티 리더가 진입할 아이템 선택
  [인벤토리 패널 열림]
   → 아이템 레어리티, 현재 층수, 예상 난이도 표시
   → 아이템 선택 시 "이 아이템으로 진입할 경우 예상 층 범위: 31~40층" 안내
  ↓
Step 3: 파티원 확인 화면
  [파티원 4인 슬롯 표시]
   → 각 파티원의 캐릭터 레벨, 주요 장비 스탯 요약 (동의 필요)
   → 파티원 준비 완료(Ready) 확인
  ↓
Step 4: 소모품 최종 체크
  [진입 전 체크리스트 팝업 — 선택적, 설정에서 비활성화 가능]
   → Mr. Gency's Exit 보유량 표시
   → 포션 보유량 표시
   → "소모품 구매 바로가기" 버튼
  ↓
Step 5: 진입 연출 → 아이템계 로드
```

**핵심 UX 포인트:**
- Step 4의 소모품 체크는 신규 플레이어를 위한 기본값 ON, 숙련 플레이어는 비활성화 가능
- 파티원의 장비 스탯 공유는 파티원 동의(opt-in) 방식. 강제 노출이 아닌 선택적 공개로 개인정보 및 플레이 방식 자율성 존중

---

### 3.3 레벨/장비 기반 매칭 필터

파티 게시판 모집 글에 다음 필터를 선택적으로 설정할 수 있어야 한다:

| 필터 항목 | 용도 | 기본값 |
|:---------|:----|:------|
| 목표 아이템 레어리티 | "Legendary 파밍" 등 명시 | 없음 (자유) |
| 최소 권장 스탯 | 해당 아이템계 난이도 기준 | 없음 |
| 경험자/초보자 | 설명 없이 빠르게 vs 가르쳐가며 | 없음 |
| 언어 | 채팅 언어 설정 | 클라이언트 언어 |
| 마이크 사용 여부 | (후기 단계, 음성채팅 구현 시) | 없음 |

**중요:** 필터 설정은 권고이지 강제 검증이 아니다. 최소 스탯 이하의 플레이어도 지원 가능하되, 모집자가 최종 결정. **자동 거절 시스템은 커뮤니티 마찰을 유발하므로 피한다.**

### 3.4 친구 목록 / 길드 시스템

#### 친구 시스템

- 게임 내 친구 추가: 캐릭터 우클릭 → "친구 추가" 또는 이름 검색
- 친구 접속 상태 상시 표시: 허브에 있는지 / 아이템계 어디에 있는지 (설정으로 가리기 가능)
- 친구의 현재 아이템계 세션 정보: "레전드리 검 아이템계 31층" → 빠른 참여 요청 버튼

#### 길드/클랜 시스템 (Phase 3 이후)

- 길드 창고: 멤버 간 아이템/소재 공유 공간. 입출고 로그 기록
- 길드 게시판: 멤버 전용 공지/모집 공간
- 길드 전용 이노센트 팜 슬롯 추가 (길드 레벨 연동)
- **길드 전용 허브 구역:** 길드 로고/색상으로 커스터마이징된 소규모 전용 공간 (FFXIV 자유 회사 하우스 경량화 버전)

## 4. 거래 시스템 UX

### 4.1 거래 모델 선택: 핵심 교훈 정리

#### 디아블로 3 실물 경매장 실패 (2012~2014)

D3 런칭 시 공식 경매장에서 실제 돈으로 거래 가능하도록 설계했다. 결과:

- 파밍 동기 완전 소멸: 최강 장비를 돈으로 즉시 구매 가능 → "왜 직접 파밍해야 하나?"
- 2년 만에 폐지. 블리자드 스스로 "우리가 한 최악의 설계 결정 중 하나"로 인정

**교훈: 거래소가 파밍을 대체하면 게임이 죽는다. 거래소는 파밍을 보완해야 한다.**

#### PoE 비공식 거래 사이트 (poe.trade → pathofexile.com/trade)

PoE는 공식 경매장이 없다. 플레이어가 아이템을 "프라이스태그"로 가격 설정 → 크롤러 사이트가 수집 → 구매자가 직접 판매자에게 귓말로 연락 → 판매자의 은신처에서 거래.

- 결과: 건강한 경제 유지
- 이유: 직접 파밍 + 크래프팅이 거래보다 비용 효율적인 경우가 많음. 거래는 "운 좋게 나온 잉여를 처분하는" 보조 수단

**교훈: 약간의 거래 마찰(직접 연락, 상대 허브 방문)이 오히려 경제 건강성을 유지시킨다.**

---

### 4.2 Project Abyss 거래 모델 권장사항

3-Space 경제 설계 문서(EndgameLoop_Economy_Research.md)에서 확인된 방향을 UX 관점에서 구체화한다.

**권장 모델: 비동기 오더북 거래소 + 직접 거래 병행**

```
[거래 방식 두 가지]

A. 거래소 (비동기 오더북)
   - 아이템 등록 → 가격 설정 → 다른 플레이어가 구매
   - 거래 수수료: 5~10% (HL Sink)
   - 등록 기간: 7일 (만료 시 자동 회수)
   - 가격 히스토리: 최근 30일 거래 기록 공개 (가격 발견 지원)
   - 구매 즉시 완료 (상대방 접속 불필요)

B. 직접 거래 (P2P)
   - 허브에서 캐릭터 우클릭 → "거래 제안"
   - 거래 창: 양쪽이 아이템/HL을 올려놓고 동시 확인 후 수락
   - 수수료 없음 (단, 양쪽 모두 접속 필요)
   - 이노센트 번들 거래, 패키지 딜 등 복잡한 거래에 적합
```

### 4.3 거래소 UI 상세

#### 검색 필터 설계

```
[기본 검색]
  └─ 아이템 이름 검색 (자동완성)
  └─ 카테고리 선택 (무기/방어구/소모품/이노센트)

[고급 필터 — 접기/펼치기 가능]
  └─ 레어리티 (Normal / Magic / Rare / Legendary / Ancient)
  └─ 스탯 최솟값 (STR, INT, DEX, VIT, SPD, LCK 각각)
  └─ 이노센트 슬롯 수 (최소)
  └─ 현재 아이템계 층수 달성 범위
  └─ 가격 범위 (최소 HL ~ 최대 HL)
  └─ 판매자 유형 (개인 / 길드 상점)

[정렬]
  └─ 가격 낮은 순 (기본)
  └─ 등록 최신 순
  └─ 스탯 합산 높은 순
```

#### 가격 발견 지원 UI

- 아이템 상세 패널에 "최근 30일 유사 아이템 거래 가격 범위" 표시
- "이 가격은 시세 대비 [저렴 / 적정 / 비쌈]" 색상 뱃지 (분석 AI 또는 백분위 기반)
- 아이템 등록 시 권장 가격 힌트 제공 (강제 아님)

### 4.4 사기 방지 UX

#### 직접 거래 사기 방지

| 사기 유형 | 방지 메커니즘 |
|:---------|:-----------|
| 유사 아이템 교체 (막판 바꿔치기) | 수락 전 3초 확인 딜레이 + "거래 내용이 변경되었습니다" 경고 |
| 0 추가 HL 사기 (1000 → 100 바꾸기) | 금액 변경 시 굵은 빨간 폰트 + 재확인 요구 |
| 이름 유사 아이템 혼동 | 레어리티 색상 + 아이템 아이콘 + 풀 네임 함께 표시 |
| 성급한 수락 방지 | 아이템 수 3개 이상 또는 HL 10,000 이상 거래 시 추가 확인 팝업 |

- 동일 이름 아이템의 스탯 편차가 클 경우 "스탯을 꼭 확인하세요" 알림 배너
- 거래 취소/환불 정책: 거래소 거래는 완료 후 취소 불가 (명시적 고지 필수)

#### Project Abyss 이모트 설계 원칙

| 원칙 | 설명 |
|:----|:----|
| 퀵슬롯 배치 | 자주 쓰는 이모트 4~8개를 단축키로 등록 |

**이모트 해금 경로:**
- 기본 이모트: 처음부터 제공 (인사, 승낙, 거절, 기다려, 도움 요청)
- 게임 내 해금: 업적, 보스 처치, 시즌 이벤트로 추가 이모트 획득
- 코스메틱 구매: 프리미엄 이모트 (기능적 차이 없음, 표현 차이만)

### 5.3 플레이어 개인 공간 (경량 하우징)

FFXIV의 하우징은 강력한 커뮤니티 형성 도구지만, 토지 부족 문제가 게임 최대 불만 사항이 되었다. Project Abyss는 이 교훈을 바탕으로 **"모든 플레이어에게 보장되는 소규모 개인 공간"** 모델을 권장한다.

#### 개인 공간 — "유물 안치소 (Relic Chamber)" 제안

- **접근:** 모든 플레이어가 처음부터 보유. 토지 경쟁 없음
- **기능:** 획득한 희귀 유물/장비 전시, 이노센트 팜 슬롯 일부, 개인 이미지 커스터마이징
- **방문 허용:** 친구/길드원에게 개방 가능 (기본값: 비공개)
- **확장:** 게임 내 활동으로 슬롯 수와 장식 종류 확대
- **PoE 은신처와의 차이:** 기능 최적화 공간이 아닌, 자신의 모험 이력을 전시하는 "박물관" 성격

이 공간은 플레이어가 게임에 대한 심리적 소유감을 갖게 하는 핵심 장치다. "내 보물들이 있는 곳"이 있다는 것은 게임을 떠날 이유를 줄인다(Sunk Cost + 감정적 애착).

### 5.4 길드 / 클랜 시스템 (Phase 3)

Phase 1~2에서는 친구 시스템과 파티 시스템으로 충분하다. 길드 시스템은 Phase 3 멀티플레이 본격 구현 시 도입한다.

**길드 핵심 기능 우선순위:**

1. 길드 채팅 채널
2. 길드 창고 (아이템/소재 공유)
3. 길드 파티 모집 우선 매칭
4. 길드 전용 이노센트 팜 슬롯 보너스
5. 길드 랭킹 (아이템계 최고 층수, 이노센트 복종 수 등)

### 5.5 채팅 시스템 (텍스트, 프리셋, 핑)

#### 채팅 채널 구조

| 채널 | 범위 | 용도 |
|:----|:----|:----|
| 일반(#general) | 허브 전체 | 자유 대화, 시장 정보 |
| 파티(#party) | 현재 파티 | 아이템계 진행 협력 |
| 귓말(DM) | 1:1 | 개인 거래 협상, 친구 소통 |
| 길드(#guild) | 길드 전체 | 길드 내부 소통 |
| 무역(#trade) | 허브 전체 | 구매/판매 공고 전용 |

#### 아이템계 내 핑 시스템

전투 중 타이핑이 어려운 상황을 위한 핑 시스템:

| 핑 유형 | 단축키 | 의미 |
|:-------|:------|:----|
| 위험 핑 | Alt+클릭 | "여기 조심해" |
| 집결 핑 | Ctrl+클릭 | "여기로 모여" |
| 이노센트 발견 | 자동 (이노센트 발견 시) | "야생 이노센트 발견" |
| 도움 요청 | 빠른 키 | "도와줘" |
| 진행 핑 | 빠른 키 | "계속 가자 / 탈출하자" |

#### 텍스트 프리셋 (Quick Chat)

- 전투 중 자주 사용하는 문구 10개를 미리 등록
- 예: "보스 조심!", "이노센트 추격 도와줘", "탈출할게요", "계속 가요"
- 접근성: 마우스/패드 조작만으로도 빠른 의사소통 가능

## 6. 허브 공간 레이아웃 원칙

### 6.1 동선 설계: 자주 쓰는 기능을 가까이

**황금 동선 원칙:** 플레이어가 허브에서 아이템계로 가는 경로에 필수 NPC(소모품 보충, 이노센트 관리)가 자연스럽게 위치해야 한다.

```
[허브 동선 계층도]

1. 진입 지점 (워프 포인트 / 귀환 지점)
   ↓ 5초 이내
2. 핵심 기능 밀집 구역
   - 창고 / 이노센트 관리소 / 대장장이
   - 소모품 상인 (Mr. Gency's Exit 포함)
   ↓ 20초 이내
3. 파티 게시판 / 파티 코디네이터
   ↓ 아이템계 진입 게이트
4. 아이템계 진입 포탈 (허브 중심축)

별도 구역 (자주 방문하지 않아도 됨):
   - 거래소 구역
   - 길드 관리 구역
   - 개인 공간 (유물 안치소)
   - 사교 광장 (이모트, 구경)
```

---

### 6.2 시각적 랜드마크로 방향 감각 제공

데스티니 2 타워의 교훈: 공간이 넓어도 방향을 잃지 않으려면 강렬한 시각적 기준점이 필요하다.

Project Abyss 허브의 랜드마크 제안:

| 랜드마크 | 위치 | 기능 |
|:--------|:----|:----|
| 심연의 문 (Abyss Gate) | 허브 중앙 | 아이템계 진입 포탈. 허브 최대 시각 요소 |
| 이노센트 수호탑 | 허브 상단 | 이노센트 팜 시설의 상징적 건축물 |
| 거래의 시계탑 | 허브 우측 | 거래소 구역 방향 지시 |
| 길드 깃발 구역 | 허브 좌측 | 현재 활성 길드들의 깃발 전시 |

허브 미니맵은 항상 화면 우상단에 고정 표시. 각 시설을 색상 코딩된 아이콘으로 표시.

---

### 6.3 인구 밀도 관리 (채널 / 인스턴스)

PSO2의 블록 시스템과 워프레임의 릴레이 채널 방식을 참조하여 웹 기반 환경에 맞게 설계한다.

#### 허브 인스턴스 설계

```
[허브 채널 시스템]

자동 배정 원칙:
- 접속 시 가장 빠른 인스턴스에 자동 배정 (핑 기준)
- 인스턴스당 최대 동시 플레이어: 50~100명 (성능 테스트 기준으로 결정)
- 길드원/친구가 다른 인스턴스에 있으면 "같은 채널로 이동" 1클릭 버튼 표시

수동 전환:
- 인스턴스 목록 UI에서 현재 인구수 표시
- "활발한 채널 (50명+)", "한산한 채널 (5명-)" 등으로 표현
- 거래소는 채널 독립 (모든 채널의 등록 물품 통합 조회)
```

### 6.4 로딩 최적화 (허브 진입 시간)

몬스터헌터 아이스본의 교훈: 로딩 화면 감소 = UX 체감 만족도 직결.

웹 기반 게임의 특수성상 브라우저 캐싱 전략이 핵심이다.

#### 웹 브라우저 환경 최적화 전략

| 전략 | 구현 방향 |
|:----|:--------|
| 허브 에셋 사전 로드 | 게임 초기 로딩 시 허브 에셋을 우선 캐싱 |
| 서비스 워커 캐싱 | 반복 방문 시 허브 리소스는 캐시에서 즉시 로드 |
| 점진적 로드 | 허브 진입 즉시 핵심 기능 영역부터 렌더링. 배경/장식 요소는 후순위 |
| 아이템계 ↔ 허브 전환 | 허브 에셋을 메모리에 유지 (가비지 컬렉션 지연). 재진입 시 로딩 최소화 |
| 목표 로딩 시간 | 허브 최초 진입: 3초 이내. 재진입(캐시): 1초 이내 |
| 동시 접속자 처리 | 허브 채널당 50~100명으로 인스턴스 분리. WebSocket 연결 부하 분산 |

**웹 기반 허브의 특수 고려사항:**

1. **브라우저 탭 비활성화:** 플레이어가 허브에 있는 동안 다른 탭으로 이동할 경우 자동 수확, 이노센트 팜 알림은 계속 작동해야 함 (Page Visibility API 활용)
2. **모바일 브라우저:** 터치 인터페이스에서 드래그 앤 드롭(이노센트 이식 등) 대안 UI 필수. 탭+확인 방식의 대안 제공
3. **메모리 한계:** 브라우저 게임은 네이티브 클라이언트 대비 메모리 제한이 엄격. 허브와 아이템계 동시 에셋 적재를 피하는 씬 전환 전략 필요

## 7. Project Abyss 허브 설계 권장사항 종합

### 7.1 허브 핵심 시설 목록 및 배치 방향

```
[허브 공간 배치도 — 권장]

                     ┌──────────────────────────────┐
                     │         [사교 광장]            │
                     │  (이모트, 구경, 잡담, 미니게임) │
                     └──────────┬───────────────────┘
                                │
        ┌──────────────────────────────────────────┐
        │                                          │
  [거래 구역]          [허브 중앙 광장]        [길드 구역]
  거래소 관리인         심연의 문 (포탈)         길드 관리인
  직접 거래 구역        이노센트 수호탑           길드 창고
  가격 시세판                                   길드 게시판
        │                    │                       │
        └────────────────────┼───────────────────────┘
                             │
              ┌──────────────┴──────────────┐
              │       [준비 구역]            │
              │   창고 / 이노센트 관리소      │
              │   대장장이 / 합성 공방        │
              │   소모품 상인 / 무기상인      │
              │   파티 게시판                │
              └──────────────┬──────────────┘
                             │
                    ┌────────┴────────┐
                    │  [진입 지점]     │
                    │  (월드 귀환 포탈) │
                    └─────────────────┘
```

**배치 원칙:**
- 아이템계 진입 포탈(심연의 문)을 중심으로
- 준비 구역(소모품, 이노센트, 파티)이 포탈 아래에서 가장 가까운 위치
- 거래 구역과 길드 구역은 양 옆으로 배치 — 필요할 때 찾아가는 곳
- 사교 광장은 위쪽 — 목적 없이 느긋하게 있을 때의 공간

---

### 7.2 아이템계 진입 UX 플로우 제안 (요약)

3.2절의 전체 플로우를 핵심 원칙으로 압축:

1. **항상 2단계 이내:** 허브 진입 → 파티 구성 → 아이템계 진입. 중간 로딩 화면 최소화
2. **소모품 마지막 체크:** 아이템계 진입 직전 Mr. Gency's Exit 보유량 표시. 신규 플레이어 보호
3. **파티원 스탯 투명성:** 아이템계 진입 전 파티원 장비 스탯 공유 (동의 기반). 역할 분담 자연 발생 유도
4. **세션 요약 복귀:** 아이템계에서 허브로 돌아올 때 세션 요약 화면 (이노센트 복종 수, 층수 달성, 처치 보스). 성취감 제공

---

### 7.3 거래소 vs 직접 거래 모델 권장

**결론: 비동기 거래소를 기본으로, 직접 거래를 보조로 제공한다.**

| 이유 | 설명 |
|:----|:----|
| D3 실물 경매장 교훈 | 거래소가 파밍을 대체해서는 안 됨. 수수료(5~10%)로 HL Sink 기능 확보 |
| PoE 교훈 | 약간의 거래 마찰이 경제를 건강하게 함. 단, 웹 게임 특성상 PoE처럼 직접 방문 모델은 적합하지 않음 |
| 접근성 | 비동기 거래소는 접속 시간대 불일치 문제 해결. 웹 기반 게임에서 특히 중요 |
| P2P 보완 | 이노센트 번들 거래 등 복잡한 협상은 직접 거래가 더 유연 |

**주의사항:**
- 이노센트는 거래소에서 거래 가능하되, 아이템계 완주 없이 구매만으로 최강 장비를 만들 수 없도록 설계 균형 필요 (게임 디자이너와 협의)
- 거래소 가격 히스토리 공개는 필수. 신규 플레이어 보호와 시장 투명성을 동시에 확보

---

### 7.4 웹 기반 허브의 특수 고려사항 종합

| 항목 | 권장 방향 |
|:----|:--------|
| 동시 접속자 관리 | 인스턴스당 50~100명. WebSocket 연결 부하 분산 아키텍처 필요 (ui-programmer 협의) |
| 로딩 시간 목표 | 허브 최초 진입 3초, 재진입 1초. 서비스 워커 캐싱 필수 |
| 모바일 터치 UI | 드래그 앤 드롭의 탭+선택 대안 필수. 이노센트 이식, 거래소 필터 터치 최적화 |
| 탭 비활성화 대응 | 팜 알림, 이노센트 수확 완료 알림을 브라우저 알림으로 전송 (허가 기반) |
| 브라우저 메모리 | 씬 전환 시 이전 씬 에셋 명시적 해제. 허브 + 아이템계 동시 적재 회피 |
| 오프라인 거래소 | 접속 중이 아니어도 거래소 등록/구매 가능. PWA 또는 웹 인터페이스 제공 검토 |

## 8. 접근성 체크리스트

Project Abyss UX 접근성 표준에 따라 허브 시스템 전반을 점검한다.

| 항목 | 허브 적용 방안 |
|:----|:------------|
| 키보드 전용 조작 | 모든 NPC 접근, 거래소 검색/필터, 이노센트 이식을 Tab + Enter + 화살표 키로 완전 조작 가능 |
| 게임패드 전용 조작 | 방향키로 허브 내 시설 간 이동. 주요 기능에 트리거 단축키 매핑 |
| 텍스트 가독성 | 거래소 가격, 이노센트 스탯 수치를 최소 12pt 이상. UI 스케일 설정 50~150% |
| 색상 의존 금지 | 레어리티 색상 외에 아이콘 형태 + 텍스트 레이블 병행. 색맹 모드 제공 |
| 이모트 접근성 | 이모트 실행 시 텍스트 설명 출력 ("가리아가 환영 인사를 합니다") |
| 채팅 텍스트 크기 | 채팅 폰트 크기 독립 조절. 배경 불투명도 조절 |
| 거래 확인 절차 | 중요 거래(고액 HL, 희귀 아이템)는 반드시 2단계 확인. 실수 방지 |
