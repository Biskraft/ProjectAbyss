## 1. 횡스크롤 절차적 생성 방식 비교

### 1-1. Spelunky — 4×4 Grid + Critical Path + Chunk

**개발사:** Mossmouth (Derek Yu) | **출시:** 2008 (Freeware), 2012 (HD), 2020 (Spelunky 2)

#### 핵심 철학

> "Always Winnable" — 어떤 시드에서도 클리어 가능한 경로가 수학적으로 보장되며, 실패의 책임은 전적으로 플레이어에게 귀속된다.

#### 생성 파이프라인

```
Phase 1: 레이아웃 생성 (Layout Generation)
  - 4×4 Room Grid 초기화 (총 16개 Room)
  - Critical Path 먼저 확정 (최상단 → 최하단)
  - 이동 방향: 좌(30%) / 우(30%) / 하(40%)
  - Grid 끝에 도달 시 강제 하강
  - Room Type 배정: Type 1(좌우), Type 2(좌우+하강), Type 3(좌우+상승)
  - 비경로 Room: Type 0 (막힘)

Phase 2: 장애물 배치 (Obstacle Placement)
  - 5×3 타일 크기의 Chunk를 Room에 삽입
  - Derek Yu가 수동 설계한 Chunk를 .lvl 파일에서 로드
  - Critical Path Room에는 반드시 통과 가능한 Chunk만 배치

Phase 3: 엔티티 배치 (Entity Placement)
  - World Config 기반 몬스터 풀에서 랜덤 배치
  - 아이템/함정 배치
```

#### 핵심 수치

| 항목 | 스펙 |
|:-----|:-----|
| Grid 크기 | 4×4 (16개 Room) |
| Room 크기 | 10×8 타일 (32px 기준 → 320×256 px) |
| Chunk 크기 | 5×3 타일 |
| 생성 단계 | 3 Phase |
| Critical Path 보장 | 수학적 완전 보장 |
| 핸드크래프트 요소 | Chunk (디자이너가 수동 제작) |

#### 장단점

| 장점 | 단점 |
|:-----|:-----|
| Critical Path의 수학적 완전 보장 | Grid가 고정(4×4)되어 구조적 단조로움 |
| 구현 복잡도 낮음 | Room 크기가 소형 → 횡스크롤 액션에 공간감 부족 |
| 디자이너 의도가 Chunk에 반영됨 | 비경로 Room(Type 0)이 많아 탐험 가치 낮음 |
| "핸드메이드 느낌" 유지 | World 간 이동 없음 (단방향 진행) |

### 1-2. Dead Cells — Concept Graph + Room Pool

**개발사:** Motion Twin / Evil Empire | **출시:** 2018

#### 핵심 철학

> "Every room feels handmade, but every run is different." — 바이옴별 50~100+ 핸드메이드 룸을 de-duplication 알고리즘으로 배치하여 매 런의 신선함을 보장하면서 퀄리티를 유지한다.

#### 생성 파이프라인

```
Phase 1: 시드 결정 & 초기화
  - 일반 런: 랜덤 시드 / Daily Challenge: 날짜 해시 고정 시드
  - BC 레벨(0~5), 보유 Rune, 바이옴 ID 초기화

Phase 2: Concept Graph 해석
  - 바이옴별 유향 그래프(Directed Graph) 로드
  - 그래프 파라미터: length(룸 수), linearity(분기 확률), density(전투 비율)
  - Rune 미보유 분기 제거 → 접근 가능 경로만 활성화
  - BC 레벨에 따른 파라미터 오버레이

  Concept Graph 예시 (Prisoners' Quarters):
  [Start] → [Combat A] → [Exploration] → [Combat B] → [Fork]
                                                         ├→ [Combat C] → [Treasure] → [Exit A]
                                                         └→ [Challenge] → [Exit B]

Phase 3: Room Assembly
  - 그래프 노드 순서대로 Room 유형 결정
  - Room Pool에서 후보 필터링 (de-duplication 포함)
  - 출입구 연결점 매칭 (방향 호환성 검사)
  - 연결 실패 시 연결 복도(Corridor) 자동 삽입

Phase 4: 몬스터 밀도 계산
  total_monsters = combat_tile_length / density_ratio
  weighted_count = SUM(normal×1 + dangerous×10)

Phase 5: PCG Fill
  - 빈 타일 자동 채움 (벽, 배경, 장식)
  - 스무딩 패스(Smoothing Pass) 적용

Phase 6: 검증 & 완성
  - 도달 가능성 검증 (모든 출구 접근 가능 여부)
  - 실패 시 연결 패치(Connection Patch) 적용
```

#### 룸 유형 분류

| 유형 | 코드 | 비율 | 설명 |
|:-----|:-----|:----:|:-----|
| Combat | CMB | 40~60% | 전투 중심 |
| Exploration | EXP | 15~25% | 플랫포밍/탐색 |
| Transition | TRN | 5~10% | 바이옴 간 연결 |
| Treasure | TRS | 5~10% | 보상 룸 |
| Challenge | CHL | 3~5% | 시간 제한 도전 |
| Secret | SCR | 5~8% | Rune 필요 비밀 룸 |
| Lore | LOR | 2~5% | 스토리/NPC |

#### 핵심 수치

| 항목 | 스펙 |
|:-----|:-----|
| 바이옴 수 | 15+ (DLC 포함) |
| 바이옴당 Room Pool | 50~100+ 핸드메이드 프리팹 |
| Concept Graph 노드 수 | 8~25개 (룸 수) |
| 타일 크기 | 16×16 px |
| Critical Path 보장 | Concept Graph 내재 보장 |
| 툴체인 | CastleDB → LDtk (오픈소스) |

#### 장단점

| 장점 | 단점 |
|:-----|:-----|
| 바이옴별 유니크한 "질감" 유지 | 바이옴당 50~100+ 룸 제작 비용 높음 |
| de-duplication으로 반복감 최소화 | Concept Graph 설계 복잡도 높음 |
| Rune 게이팅으로 탐험 공간 단계적 확장 | 초기 진입 장벽 (툴체인 학습 필요) |
| PCG Fill로 빈 공간 자동 처리 | 완전 랜덤 생성 대비 다양성 제한 |
| BC 난이도 연동 파라미터 스케일링 | |

### 1-3. Rogue Legacy 2 — Castle Blueprint + Room Shuffle

**개발사:** Cellar Door Games | **출시:** 2022

#### 핵심 철학

메트로베니아의 고정 맵 감성을 유지하면서도 로그라이크의 리플레이성을 확보하는 독특한 하이브리드 접근이다. "Castle Blueprint"라는 개념으로 각 구역의 전반적인 형태(고성, 동굴, 탑 등)는 고정하되, 구역 내부의 Room 배치를 런마다 섞는다.

#### 구조 특징

```
매크로 구조 (고정):
  - 7개 구역: Citadel Agartha, Axis Mundi, The Stygian Study,
    The Kerguelen Plateau, The Dry Lake Bed, The Pishon Dry Lake, Tulia's Palace
  - 각 구역의 전체 형태와 분위기 고정
  - 구역 간 연결 구조 고정

마이크로 구조 (랜덤):
  - 구역 내 Room 배치는 런마다 다름
  - 각 Room은 핸드크래프트 프리팹
  - Room 연결 방식과 순서가 랜덤하게 조합됨

영구 진행 시스템 (메타 레이어):
  - 적 처치로 골드 획득
  - 허브에서 캐릭터 스탯 영구 업그레이드
  - 업그레이드 없이 후반 구역은 실질적으로 진행 불가 (스탯 소프트 게이트)
```

#### 특징적 설계

- **Architect 시스템:** 골드를 지불하면 현재 런의 맵을 고정 (다음 런 도전 시 동일 맵 보장). 특정 맵을 마스터하려는 플레이어에게 선택지 제공.
- **Heirloom 시스템:** 구역 보스 처치 시 해금되는 능력이 영구 적용 → 메트로베니아 게이트 구조의 영구 진행 버전.
- **Class 다양성:** 매 런 무작위 클래스 3종 중 선택 → 조작감 변화로 같은 맵도 다르게 체험.

#### 장단점

| 장점 | 단점 |
|:-----|:-----|
| 핸드크래프트 Room의 퀄리티 유지 | Room Shuffle만으로 신선함 한계 존재 |
| 메트로베니아 분위기 유지 | 완전 절차적 대비 랜덤성 낮음 |
| 영구 성장으로 리텐션 확보 | 초반 진입 장벽 (스탯 소프트 게이트) |
| Architect 시스템의 유연한 난이도 조절 | |

### 1-4. Sundered — 핸드크래프트 허브 + 절차적 연결

**개발사:** Thunder Lotus Games | **출시:** 2017

#### 핵심 철학

코어 내러티브 공간(허브, 보스 방, 핵심 능력 방)은 완전 핸드크래프트로 제작하고, 허브를 연결하는 통로와 적 조우 구역만 절차적으로 생성하는 "허브 앵커(Hub Anchor)" 전략이다.

#### 구조 특징

```
핸드크래프트 구역 (고정):
  - 시작 허브 (Sanctuary)
  - 3개 주요 허브 (각 지역의 앵커 포인트)
  - 보스 방 전체
  - 핵심 능력 획득 방

절차적 구역 (랜덤):
  - 허브 사이를 연결하는 통로
  - 적 조우가 발생하는 사이드 구역
  - 랜덤 선택된 사전 제작 "섹션" 조합
```

#### 핵심 시사점

- **앵커 포인트 방식:** 핵심 내러티브/기계적 모먼트는 완전 고정, 그 사이 공간만 랜덤화. 이로써 스토리 플레이와 절차적 탐험이 충돌하지 않음.
- **엘드리치(Eldritch) 적 밀물:** 특정 구역 진입 시 대량의 적이 절차적으로 스폰되는 "적 밀물(Enemy Tide)" 메커니즘. 절차적 공간의 개성을 강화하는 방식.
- **단점:** 절차적 구역이 "허브 사이를 채우는 공간"으로 느껴지기 쉬워 탐험 동기가 약화될 수 있음. Dead Cells 대비 절차적 공간의 개성 부족.

### 1-5. Hades — Room Graph + Encounter Pool

**개발사:** Supergiant Games | **출시:** 2020

#### 핵심 철학

> "절차적 생성 없이 로그라이크를 만든다." — Hades는 실제로 물리적인 레벨 레이아웃 자체는 핸드크래프트이다. 절차적 요소는 "어떤 방에서 어떤 조우(Encounter)를 경험하는가"에만 적용된다.

#### 구조 특징

```
완전 핸드크래프트 레이아웃:
  - Tartarus, Asphodel, Elysium, Temple of Styx 각 구역의 모든 방 수동 제작
  - 각 방의 형태, 지형, 출입구 위치 전부 고정

절차적 요소 (Encounter Pool):
  - 방 입장 시 해당 구역의 Encounter Pool에서 조우 선택
  - Encounter = 적 구성 + 배치 패턴의 조합
  - 구역별 Encounter Pool: 40~60개 조합

Room Graph (방 경로):
  - 각 방에서 다음 방으로의 연결 구조는 Run마다 다름
  - 플레이어는 이전 방의 보상 종류를 보고 다음 방 선택
  - 선택지: 하트, 다크티스, 잠쇠 등 보상 유형 표시
```

#### Boon 시스템과의 연계

- 올림포스 신의 Boon(축복)은 런마다 랜덤 제공
- 빌드(Build)의 방향성이 Boon 조합에 따라 달라짐
- 물리적 레이아웃이 고정이어도 "빌드 경험"의 다양성으로 리플레이성 확보

#### 핵심 시사점

- **"레벨 레이아웃 절차화 없이도 리플레이성 확보 가능"**: Encounter Pool과 보상 경로 선택만으로 충분한 다양성 제공.
- **내러티브 밀도 최대화**: 고정 레이아웃이기에 모든 방에 NPC 대화, 환경 스토리텔링 등 내러티브 요소를 치밀하게 배치 가능.
- **한계**: 플레이 시간이 길어질수록(200시간+) 맵 암기가 진행되어 탐험감 소멸. Dead Cells 대비 재방문 신선함 낮음.

## 2. 핸드크래프트 + 절차적 혼합 전략

### 2-1. "매크로는 핸드크래프트, 마이크로는 절차적" 원칙

이 원칙은 사실상 모든 성공적인 하이브리드 게임에서 공통적으로 관찰되는 황금률이다.

```
매크로 (Macro) = 구조적 골격
  - 구역(Zone)의 위치와 개수
  - 구역 간 연결 관계
  - 핵심 내러티브 이벤트의 위치
  - 보스 방, 세이브 포인트, 워프 포인트 위치
  - "이 세계가 어떻게 생겼는가"에 대한 답

마이크로 (Micro) = 내부 내용물
  - 구역 내부의 Room 배치와 순서
  - Room 내 지형 구성 (Chunk/Section 조합)
  - 적의 종류와 배치
  - 아이템/함정/보상 위치
  - "이 구역 안에서 무엇을 만나는가"에 대한 답
```

#### 원칙이 작동하는 이유

**플레이어의 인지 부하(Cognitive Load) 관점에서:**
- 매크로가 고정되면 플레이어는 "대략 어디로 가야 하는지" 기억할 수 있다. 공간적 기억(Spatial Memory)이 형성되어 세계에 대한 친밀감이 생긴다.
- 마이크로가 랜덤이면 세부적인 장애물은 항상 새롭다. "어떤 적이 나올지 모른다"는 긴장감이 유지된다.

**SDT(자기결정이론) 관점에서:**
- 매크로 고정 → 유능감(Competence): 길을 알고 있다는 숙달 감각 유지
- 마이크로 랜덤 → 자율성(Autonomy): 매번 다른 상황에서의 선택과 대처
- 이 두 축의 균형이 장기 플레이 동기 유지의 핵심

### 2-2. 어떤 부분이 고정이고 어떤 부분이 랜덤인가

| 핵심 능력 획득 방 | 튜토리얼 챔버로 기능 |

| 요소 | 처리 방식 |

### 2-4. "핸드메이드 느낌"을 유지하는 조건

절차적 생성이 "기계가 만든 것처럼 느껴지는" 이유와 그 해결책:

#### 조건 1: 충분한 Chunk/Room Pool 크기

```
최소 기준:
  - 구역당 Chunk Pool: 40개 이상 (비복잡 구역)
  - 구역당 Chunk Pool: 50개 이상 (최종 구역)
  - Room Pool: 바이옴당 50개 이상 (Dead Cells 기준)

이유: Pool 크기가 작으면 같은 패턴이 반복되어 플레이어가 알아챔.
     연속 N회 미사용 룸 우선 선택(de-duplication) 알고리즘을 함께 사용.
```

#### 조건 2: Chunk의 디자이너 직접 제작

```
Derek Yu의 원칙: Chunk는 반드시 디자이너가 플레이해보며 수동 설계.
- 통과 가능성 직접 검증
- 전투 공간 / 탐색 공간 / 보상 공간의 의도적 배치
- "이 Chunk에서 어떤 플레이어 행동이 유도되는가?" 의문 필수
```

#### 조건 3: Context-Aware 배치

```
잘못된 방식: 완전 랜덤 선택
올바른 방식:
  - 직전 Chunk와 출입구 방향 호환성 검사
  - 연속된 같은 종류의 장애물 배치 제한
  - 전투 후에는 탐색/보상 공간 배치 가중치 증가
  - 구역 초반에는 쉬운 Chunk, 후반에는 어려운 Chunk 선택 가중치 조정
```

#### 조건 4: 구역별 시각 언어 일관성

```
각 구역(바이옴)의 Chunk는 해당 구역의 시각적 테마에 100% 부합해야 함.
예: 빙결 동굴 Chunk에 화염 원소 장치 배치 금지.
   지하 수로 Chunk는 수류/물 메커니즘이 포함된 것 위주.
```

#### 조건 5: 특수 Chunk의 희소성

```
바이옴 정체성을 표현하는 "특수 Chunk"는 구역당 3~5종 제작.
이 특수 Chunk는 낮은 확률(5~10%)로만 등장 → 희소성으로 인한 "발견의 기쁨".
예: 왕좌실 Chunk (중앙 성채), 실험실 폭발 Chunk (마법 연구소)
```

## 3. Room Template 설계 원칙

### 3-1. Room 크기와 형태 표준화

ECHORIS의 현행 스펙(System_World_ProcGen.md 기준)과 레퍼런스 비교:

| 게임 | Room 크기 (타일) | 타일 크기 (px) | Room 크기 (px) |
|:-----|:---------------|:-------------|:--------------|
| Spelunky | 10×8 | 32×32 | 320×256 |
| Dead Cells | 가변 | 16×16 | 바이옴별 상이 |
| ECHORIS (현행) | 60×34 | 16×16 | 960×544 |

#### 현행 스펙 평가

ECHORIS의 960×544 px Room은 Spelunky 대비 약 6배 큰 공간이다. 이는 횡스크롤 액션 RPG의 전투 공간 요구사항(적 다수, 다방향 공격 패턴)에 적합하다. 단, Room이 너무 크면 절차적 생성의 단조로움이 증폭되어 Chunk Pool 요구량이 증가한다.

#### 권장 표준화 원칙

```
기본 Room 크기: 60×34 타일 (현행 유지)
  이유: 횡스크롤 전투 공간 충분, 플랫포밍 다이나믹 가능

예외 Room 크기:
  - 보스 방: 120×68 타일 (기본의 2배) — 고정 핸드크래프트
  - 이벤트/내러티브 방: 40×34 타일 (기본의 2/3) — 고정 핸드크래프트
  - 통로 방(Corridor): 20×20 타일 — 절차적 생성

Room 형태 유형:
  - 수평 기본형: 60×34 (가장 일반적)
  - 수직 통로형: 20×50 (벽 타기 구간, 천공의 탑 전용)
  - 수중 동굴형: 60×50 (지하 수로 전용)
```

### 3-2. 출입구(Connection Point) 규격

#### 출입구 표준 규격

```
출입구 폭: 4타일 (64px) — 캐릭터 이동 중 충분한 여유
출입구 위치: Room 가장자리에서 정렬된 위치
출입구 방향: 상(U), 하(D), 좌(L), 우(R) 4방향
```

#### Room Type과 출입구 조합

| Type | 이름 | 출입구 | 용도 |
|:----:|:-----|:-------|:-----|
| 0 | Dead End | 없음 | 비경로 막힌 방 (비밀 방으로 변환 가능) |
| 1 | 수평 통로 | L + R | 수평 Critical Path |
| 2 | 하강 분기 | L + R + D | Critical Path의 하강 지점 |
| 3 | 상승 분기 | L + R + U | 보상/선택 경로 상승 |
| 4 | 수직 통로 | U + D | 천공의 탑 전용 수직 구간 |
| 5 | 십자 분기 | L + R + U + D | 복잡한 분기점 (희소) |

#### 출입구 연결 검사 알고리즘

```
연결 유효성 검사:
1. 방향 호환성: Room A의 R 출구 → Room B의 L 입구 (방향 매칭)
2. 높이 호환성: 출입구 y좌표 차이 ≤ 2타일 (±32px)
3. 연결 실패 시: 연결 복도(Corridor Room) 자동 삽입
4. 복도 삽입 한계: 구역 내 최대 20% Room을 복도로 채울 수 없음
```

## 4. 타일 기반 생성 알고리즘

### 4-1. Wave Function Collapse (WFC)

#### 개요

WFC는 입력 패턴에서 타일 간 "이웃 관계 규칙"을 추출하여, 이를 기반으로 국소적으로 일관된 출력을 생성하는 알고리즘이다. Maxim Gumin이 2016년 제안했으며, 마르코프 랜덤 필드(Markov Random Field)의 한 응용이다.

#### 작동 원리

```
입력: 샘플 타일맵 (디자이너가 직접 제작한 소형 맵)
처리:
  1. 샘플에서 모든 타일 쌍의 인접 관계 추출 (상하좌우)
  2. 출력 그리드 초기화: 모든 셀에 모든 타일 가능 (Superposition)
  3. 엔트로피 최소 셀 선택 (가장 결정에 가까운 셀)
  4. 해당 셀의 타일 결정 (Collapse)
  5. 이웃 셀에 제약 전파 (Propagation)
  6. 모순 발생 시 백트래킹 또는 재시작
출력: 샘플과 "같은 규칙"을 따르지만 새로운 타일맵
```

#### 횡스크롤 게임에서의 적용 가능성

| 적용 영역 | 가능성 | 주의사항 |
|:---------|:------:|:---------|
| 배경 환경 타일 (벽, 바닥 장식) | 높음 | 통과 불가 타일은 규칙에 포함 |
| 동굴 형태 지형 생성 | 높음 | 연결성 보장 별도 검증 필요 |
| 실내 방 레이아웃 | 중간 | 출입구 위치 고정 후 내부 생성 |
| Critical Path 보장 | 낮음 | WFC 자체는 경로 보장 미지원 |

#### ECHORIS에서의 활용 제안

```
추천 활용 범위: 시각적 배경 타일 자동 채움 (Dead Cells의 PCG Fill 역할)
  - Chunk 배치 후 빈 배경 공간을 WFC로 자동 채움
  - 예: 동굴 벽 패턴, 석재 무늬, 물 흐름 배경 등

비추천 활용 범위: Room 레이아웃 자체 생성, 적 배치
  - Critical Path 보장 불가
  - 실패 시 복잡한 백트래킹 처리 필요
  - 실시간 생성 퍼포먼스 이슈 (PixiJS 환경)
```

---

### 4-2. Cellular Automata (세포 자동자) — 동굴 생성

#### 개요

Conway의 Game of Life를 기반으로 한 지형 생성 기법이다. 각 셀이 이웃 셀의 상태에 따라 벽(1)/공간(0)으로 반복 갱신되어 자연스러운 동굴 형태가 형성된다.

#### 알고리즘

```
초기화: 각 셀을 확률 p로 벽(1) 또는 공간(0)으로 랜덤 채움 (p ≈ 0.45)
반복 (4~6회):
  - 각 셀의 8방향 이웃 중 벽의 수를 계산
  - 이웃 벽 수 ≥ 5이면 → 벽으로 전환
  - 이웃 벽 수 < 5이면 → 공간으로 전환
후처리:
  - 연결 컴포넌트 분석 (BFS/DFS)
  - 고립된 동굴 구역 제거 또는 연결
  - Critical Path 영역 보호 (강제 공간으로 유지)
```

#### ECHORIS 적용 가능 영역

```
적합: 빙결 동굴, 심연의 구, 지하 수로의 자연 지형 구간
  - 불규칙한 동굴 벽면 생성에 탁월
  - 낮은 계산 비용, 빠른 생성 속도

주의: Chunk 기반 방식과 혼합 시 접합 처리 필요
  - CA 생성 지형과 핸드크래프트 Chunk의 경계에 블렌딩 필요
  - 출입구 주변 타일은 CA 생성 금지 구역으로 설정
```

### 4-3. BSP (Binary Space Partition) — 던전 구획

#### 개요

공간을 재귀적으로 이분할하여 방을 배치하는 알고리즘이다. 고전적인 로그라이크(NetHack, ADOM)에서 광범위하게 사용되었다.

#### 알고리즘

```
1. 전체 공간을 하나의 직사각형으로 시작
2. 무작위 지점에서 수직 또는 수평으로 분할
3. 최소 크기 도달 시까지 반복 분할
4. 각 최종 잎 노드(Leaf)에 방 생성 (잎보다 약간 작게)
5. 방들을 트리 구조를 따라 복도로 연결
6. 연결 시 Critical Path 보장 (트리 구조 자체가 연결성 보장)
```

#### 장단점

| 장점 | 단점 |
|:-----|:-----|
| 연결성 수학적 보장 (BSP 트리 구조상) | 방이 모두 직사각형 → 유기적 형태 어려움 |
| 방 크기 분포 제어 용이 | 복잡한 분기 구조 표현 한계 |
| 구현 단순 | 횡스크롤에서 수직 분할이 부자연스러울 수 있음 |

#### ECHORIS 적용 가능성

```
제한적 활용 가능: 마법 연구소, 중앙 성채의 실내 구획 생성
  - 직사각형 방이 어울리는 성 내부/실험실 환경에 적합
  - Room Grid 내부의 서브 구획 생성에 활용 가능

비추천: 전체 World 레이아웃 생성
  - 이미 Room Grid 방식(스펠렁키 참고)을 채택하고 있으므로 중복
  - 유기적 지형(동굴, 수로)에는 부적합
```

### 4-4. Marching Squares — 지형 스무딩

#### 개요

타일 기반 지형의 계단식 모서리를 부드럽게 처리하는 기법이다. 이진 격자(Binary Grid)를 폴리곤 경계선으로 변환하여 곡선 지형을 표현한다.

#### 횡스크롤에서의 활용

```
활용 시나리오:
  - Cellular Automata로 생성된 동굴 지형의 벽면 스무딩
  - 타일 경계를 45도/곡선 타일로 자동 대체
  - 비주얼 퀄리티 향상 (계단 현상 제거)

구현 주의사항:
  - 충돌 검사(Collision Detection)는 원본 이진 그리드 기반 유지
  - 시각적 스무딩이 게임플레이 물리에 영향 주지 않도록 레이어 분리
  - PixiJS: Graphics API로 폴리곤 렌더링 가능, 단 대량 사용 시 퍼포먼스 주의
```

## 5. Critical Path 보장 메커니즘

### 5-1. Spelunky 방식: 경로 먼저 확정 후 채움

이는 "Always Winnable"을 수학적으로 보장하는 가장 강력한 접근이다.

#### 알고리즘 상세

```
Step 1: Grid 초기화
  - N×M Room Grid 생성 (모든 Room 비어있음)
  - 시작 Room: 최상단 행의 랜덤 Column 선택

Step 2: Critical Path 확정
  현재 Column 위치를 추적하며:
    a. 이동 방향 결정: 좌(30%), 우(30%), 하(40%)
    b. Grid 경계 도달 시: 강제 하강
    c. 하강 시: 현재 Room에 하강 출입구(D) 추가
    d. 최하단 행 도달 시: 경로 완성

  경로 품질 검사:
    - 경로 길이 ≥ Grid 전체 셀 수의 40% 미만 → 재시작
    - 이유: 너무 짧은 경로는 탐험감 부족

Step 3: Critical Path Room Type 배정
  - 직선 구간: Type 1 (L+R)
  - 하강 전환점: Type 2 (L+R+D)
  - 상승 분기(보상 경로): Type 3 (L+R+U)

Step 4: 비경로 Room 채움
  - 비경로 Room: Type 0 (막힘) 또는 Type 1 (탐험 경로)
  - 비경로 Type 1은 비밀 방, 보상 방으로 활용

Step 5: Critical Path Room에 통과 가능 Chunk만 배치
  - 각 Chunk는 태그 보유: [CP_SAFE = true/false]
  - CP_SAFE = true Chunk만 Critical Path에 배치
  - 비경로 Room에는 모든 Chunk 배치 가능
```

#### 장점

- 완전한 수학적 보장: 경로가 먼저 확정되므로 클리어 불가 상황 원천 차단
- "실패 = 플레이어 탓" 원칙 유지 가능
- 구현 단순 (재귀 불필요)

#### 단점

- Grid 크기가 고정되어 구조적 다양성 제한
- 경로가 지나치게 단순할 수 있음 (지그재그 패턴 반복)
- 수직 이동이 많은 메트로베니아 구조 표현에 한계

### 5-2. Dead Cells 방식: Concept Graph 내재 보장

#### 접근법

Critical Path를 "명시적으로 먼저 계산"하지 않고, Concept Graph 구조 자체가 연결성을 보장한다.

```
보장 원리:
  - Concept Graph는 유향 비순환 그래프(DAG)로 정의
  - [Start] → ... → [Exit] 경로가 그래프 정의에서 반드시 존재
  - Room Assembly 후 출입구 연결 검사 실행
  - 연결 실패 시: 연결 복도 자동 삽입 (패치 방식)

검증 단계 (Phase 6):
  BFS로 Start → Exit 도달 가능성 검사
  실패 시: 연결 패치 적용 (최단 연결 복도 삽입)
  패치 불가 시: 레벨 재생성
```

#### 스펠렁키 방식과의 비교

| 항목 | Spelunky | Dead Cells |
|:-----|:--------:|:----------:|
| 보장 시점 | 생성 전 (선제적) | 생성 후 (사후 검증) |
| 보장 강도 | 수학적 완전 보장 | 높은 확률 + 패치 |
| 구현 복잡도 | 낮음 | 중간 |
| 경로 다양성 | 낮음 (격자 제약) | 높음 (그래프 분기) |
| 재생성 빈도 | 없음 (보장됨) | 드물게 발생 |

### 5-3. ECHORIS Critical Path 알고리즘

System_World_ProcGen.md의 현행 알고리즘에 추가 권장사항:

#### 현행 알고리즘 강화 제안

```
현행: 스펠렁키식 Critical Path Grid
  좌(30%) / 우(30%) / 하(40%) 이동
  경로 길이 ≥ Grid 셀 수의 40%

추가 권장 1 — 최소 하강 횟수 보장:
  구역 Grid 높이의 60% 이상을 하강해야 함
  예: 4×3 Grid → 최소 2회 하강 보장
  이유: 지나치게 수평적인 레이아웃 방지

추가 권장 2 — 방향 다양성 검사:
  Critical Path에서 같은 방향 연속 3회 이상 금지
  예: R, R, R → 재생성
  이유: 단조로운 일직선 경로 방지

추가 권장 3 — 분기점 최소 보장:
  구역 Grid 크기에 따라 최소 분기점(선택 경로) 수 보장
  예: 4×4 Grid → 최소 2개의 선택 경로 Room(Type 3) 보장
  이유: 탐험 요소 확보
```

#### 게이트 검증 알고리즘 (현행 유지 + 강화)

```
검증 1: BFS로 시작 → 최종 보스 도달 가능성
검증 2: Critical Path 위의 모든 게이트가 해당 시점 플레이어 능력으로 통과 가능한지
  → 능력 게이트: Critical Path 위에 배치 불가 (절대 규칙)
  → 스탯 게이트: 비경로에만 배치 (절대 규칙)
검증 3: 폴백 맵 — 연속 10회 재시드 실패 시 사전 검증 완료 폴백 맵 사용
```

### 5-4. 플레이어빌리티 검증 알고리즘

Critical Path 검증을 넘어, 게임플레이 가능성 자체를 검증하는 추가 알고리즘:

```
점프 도달 가능성 검사:
  - 각 Chunk 내 플랫폼 간 최대 수직 간격 ≤ 캐릭터 점프 높이
  - 이단 점프 전에는 이단 점프 없이 통과 가능한 Chunk만 배치
  - 데이터: 각 Chunk에 required_abilities[] 배열 태그 부여

공간 충분성 검사:
  - Room 내 캐릭터가 이동 가능한 최소 연속 공간 보장
  - 좌우 이동: 최소 10타일 연속 공간
  - 상하 이동: 최소 5타일 연속 공간

전투 공간 검사:
  - 전투 Room에서 적이 스폰될 수 있는 최소 공간 보장
  - 동시 스폰 예정 적 수 × 최소 개인 공간 ≤ Room 사용 가능 공간
```

## 6. 월드 vs 아이템계 생성 차이

### 6-1. 두 시스템의 철학적 차이

| 항목 | 월드 (World) | 아이템계 (Item World) |
|:-----|:------------|:--------------------|
| **목적** | 탐험, 발견, 내러티브 | 파밍, 강화, 야리코미 |
| **기대 수명** | 시즌 단위 (3~6개월) | 매 진입마다 새로 생성 |
| **시드 수명** | 장기 고정 | 즉시 생성, 퇴장 시 소멸 |
| **플레이어 기억** | "대략 어디에 뭐가 있다" 기억 형성 | 기억 필요 없음, 매번 새로움 |
| **내러티브 밀도** | 높음 (이벤트, NPC, 환경 스토리) | 낮음 (전투/파밍 집중) |
| **Critical Path** | 구역 내 + 구역 간 전체 검증 | 지층별 독립 검증 |
| **핸드크래프트 비율** | 50% (매크로 완전 고정) | 0% (완전 절차적) |

---

### 6-2. 월드 생성의 특수성 — "시즌 리셋 친화적" 설계

월드는 3~6개월마다 새로운 시드로 재생성된다. 이는 다음 조건을 충족해야 한다:

```
불변 요소 (어떤 시드에서도 동일):
  - 7개 구역의 위치와 상호 연결 관계
  - 각 구역의 바이옴과 테마
  - 보스 방의 존재와 위치
  - 능력 게이트의 논리적 순서

변동 요소 (시즌마다 다름):
  - 구역 내 Room Grid의 구체적 배치
  - Room 내 Chunk 조합
  - 적 종류와 배치
  - 아이템/비밀 방 위치
  - 환경 장식 세부
```

이 방식은 플레이어가 "세계의 구조는 알고 있지만, 세부 내용은 다시 탐험해야 한다"는 경험을 제공한다.

---

### 6-3. 동일 엔진으로 두 모드를 지원하는 설계

두 시스템을 별도 엔진으로 구현하면 코드 중복과 유지보수 비용이 증가한다. 공통 파이프라인으로 통합 가능하다.

#### 통합 LevelGenerator 인터페이스 설계

```typescript
// 공통 인터페이스
interface LevelGeneratorConfig {
  mode: 'world' | 'itemworld';
  seed: number;
  gridSize: { cols: number; rows: number };
  biome: BiomeID;
  difficulty: 1 | 2 | 3 | 4 | 5;

  // 모드별 특수 파라미터
  worldConfig?: {
    zoneID: WorldZoneID;
    seasonSeed: number;
    isStartZone: boolean;
  };
  itemWorldConfig?: {
    itemID: string;
    itemLevel: number;
    strataDepth: number;     // 기억의 지층 깊이
    rarityTier: RarityTier;
    partySize: 1 | 2 | 3 | 4;
  };
}

// 공통 파이프라인 (두 모드 모두 사용)
class LevelGenerator {
  generate(config: LevelGeneratorConfig): LevelData {
    const grid = this.initGrid(config);            // Phase 1
    const path = this.buildCriticalPath(grid);     // Phase 2
    const typed = this.assignRoomTypes(grid, path); // Phase 3
    const chunked = this.assembleChunks(typed, config.biome); // Phase 4
    const populated = this.placeEntities(chunked, config);    // Phase 5
    const validated = this.validate(populated, config);        // Phase 6
    return validated;
  }
}
```

#### 모드별 차이점 처리

| Phase | 월드 처리 | 아이템계 처리 |
|:------|:---------|:------------|
| 시드 결정 | 서버 고정 시드 (계절 단위) | `hash(itemID + level + strataDepth)` |
| Grid 크기 | 구역별 고정 (3×3~5×5) | 4×4 고정 (레어리티별 지층 수로 차별화) |
| Chunk Pool | 바이옴별 풀 | 아이템 속성 + 바이옴별 풀 |
| 적 배치 | 구역 난이도 기반 | 아이템 레벨 + 지층 깊이 기반 |
| 이노센트 배치 | 없음 | 아이템 내 이노센트 목록 기반 |
| 환경 효과 패널 | 일부 특수 구역 | 지층별 환경 효과 적극 활용 |
| 게이트 | 능력 게이트 + 스탯 게이트 | 스탯 게이트만 |
| 검증 | 전체 월드 연결성 검증 | 지층별 독립 검증 |

---

### 6-4. 월드에서 재방문 시 미세 변화 처리

메트로베니아 특성상 플레이어는 같은 구역을 능력 획득 후 반복 방문한다. 절차적 생성이 이를 지원해야 한다.

```
재방문 구역의 변화 설계:
  1. 기본 Room Grid는 시즌 동안 고정 유지 (같은 레이아웃)
  2. 플레이어 레벨/능력에 따른 엘리트 적 스폰 확률 증가
  3. 비밀 방 내용물 갱신 (주간 리셋)
  4. 일부 Chunk의 계절적 변화 (선택적):
     - 빙결 동굴: 얼음 두께 변화 (시각적)
     - 지하 수로: 수위 변화 (일부 경로 개폐)
     - 천공의 탑: 날씨 패턴 변화 (시각/음향)
  5. 신규 출현 희귀 이노센트 (주간 이벤트)
```
