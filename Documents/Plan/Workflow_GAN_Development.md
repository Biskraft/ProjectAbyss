# ECHORIS GAN 개발 워크플로우

> Generator-Discriminator 적대적 루프를 개발 프로세스에 적용한 워크플로우.
> 생성과 검증을 분리하여, 한 쪽이 만들고 다른 쪽이 깨뜨리는 구조로 품질을 끌어올린다.

---

## 핵심 원칙

```
Generator (생성)  →  Discriminator (검증)  →  피드백  →  재생성  →  ...통과까지 반복
```

- Generator는 최선을 다해 만든다
- Discriminator는 최선을 다해 깨뜨린다
- 둘 다 같은 목표(게임 품질)를 위해 싸운다
- 통과 기준을 사전에 정의한다 (Pass Criteria)
- 3회 반복 후에도 미통과 시 스코프 축소 또는 보류

---

## 역할 매핑

| GAN 역할 | 터미널 | 담당 | 원칙 |
|---------|--------|------|------|
| **Generator** | DEV / RES / Designer | 코드, 문서, 설계를 생성 | 빠르게, 완성도보다 의도 전달 우선 |
| **Discriminator** | PM / QA / Reviewer | 결과물을 검증하고 문제를 찾는다 | 비평적, 구체적, 대안 제시 |
| **Victor** | 최종 판정 | Pass/Fail 결정 | Discriminator도 틀릴 수 있다 |

## 워크플로우 파이프라인

### 1. GDD 파이프라인 (기획 문서)

```
[G] Designer: GDD 초안 작성
    ↓
[D] PM: 정합성 검증
    - 다른 GDD와 수치/용어 충돌?
    - 스파이크 정렬 여부? ("아이템계를 강화하는가?")
    - 구현 가능성?
    - 5단계 구조 준수?
    ↓
[F] 피드백: 구체적 문제 목록 + 수정 방향
    ↓
[G] Designer: 수정본 작성
    ↓
[D] PM: 재검증
    ↓
Pass → 문서 확정, Document_Index 등록
Fail (3회) → 스코프 축소 또는 보류
```

**Pass Criteria:**
- 다른 GDD와 수치 충돌 0건
- 스파이크 정렬 질문 통과
- 구현 난이도 Tier 0-2 이내
- 용어집(Glossary) 미등록 용어 0건

---

### 2. 코드 파이프라인 (구현)

```
[G] DEV: 기능 구현
    ↓
[D] PM: 코드 리뷰 (읽기 전용)
    - 태스크 브리프와 일치하는가?
    - GDD 스펙과 수치가 맞는가?
    - 기존 시스템과 충돌하는가?
    ↓
[D] QA: 플레이 테스트
    - 의도한 대로 작동하는가?
    - 엣지 케이스에서 깨지는가?
    - 게임 느낌(feel)이 맞는가?
    ↓
[F] 피드백: 버그 목록 + 느낌 리포트
    ↓
[G] DEV: 수정
    ↓
[D] QA: 재테스트
    ↓
Pass → 커밋, 배포
Fail (3회) → 기능 축소 또는 다른 접근법
```

**Pass Criteria:**
- 태스크 브리프 항목 100% 충족
- 크래시/프리즈 0건
- GDD 수치와 오차 5% 이내
- 히트스탑/넉백 등 피지컬 느낌 Victor 승인

---

### 3. 리서치 파이프라인 (조사/분석)

```
[G] RES: 리서치 문서 작성
    ↓
[D] Designer: 적용 가능성 검증
    - ECHORIS에 실제로 쓸 수 있는가?
    - 스파이크와 충돌하지 않는가?
    - 액션 아이템이 명확한가?
    ↓
[F] 피드백: "이건 쓸 수 있다 / 이건 안 맞다 / 이건 변형 필요"
    ↓
[G] RES: 액션 아이템 정리, GDD 반영 제안
    ↓
Pass → Research 폴더 등록, 관련 GDD에 참조 추가
Skip → 참고용으로만 보관
```

**Pass Criteria:**
- 액션 아이템 최소 1건
- ECHORIS 적용 시 스파이크 강화 여부 명시
- 출처/근거 포함

---

### 4. 비주얼/에셋 파이프라인

```
[G] Artist: 에셋 제작 (컨셉, 스프라이트, 타일셋)
    ↓
[D] Art Director: 톤 & 매너 검증
    - 고딕 다크 판타지 톤에 맞는가?
    - 해상도 640x360 기준 가독성?
    - 기존 에셋과 스타일 일관성?
    ↓
[D] DEV: 기술 검증
    - 스프라이트 시트 규격 준수?
    - 애니메이션 프레임 수 적정?
    - 메모리 예산 내?
    ↓
[F] 피드백
    ↓
[G] Artist: 수정
    ↓
Pass → assets/ 등록, 코드 연동
```

---

### 5. 밸런스 파이프라인 (수치 조정)

```
[G] Designer: 수치 테이블 작성 (CSV/Sheets)
    ↓
[D] Simulator: 시뮬레이션 검증
    - DPS 커브가 의도대로인가?
    - 레어리티 간 격차가 적절한가?
    - 게이트 돌파 시점이 자연스러운가?
    ↓
[D] QA: 실제 플레이 검증
    - 숫자가 "느낌"으로 맞는가?
    - 너무 쉽거나 너무 어려운 구간?
    ↓
[F] 피드백: 구간별 체감 리포트
    ↓
[G] Designer: 수치 조정
    ↓
Pass → Sheets/ SSoT 갱신, 코드 반영
```

## 적대적 루프 규칙

### 3-Strike Rule

- 동일 이슈로 3회 반복 시 더 이상 수정하지 않는다
- 선택지: 스코프 축소 / 다른 접근법 / 보류
- "완벽하게 만들기"가 아니라 "통과 기준을 넘기기"가 목표

### Discriminator 의무

- "안 된다"만 말하지 않는다. 반드시 **왜 안 되는지 + 대안**을 제시한다
- 감정이 아닌 기준(Pass Criteria)으로 판단한다
- Generator의 의도를 먼저 이해한 후 비평한다

### Generator 의무

- 피드백을 무시하지 않는다
- 동의하지 않으면 근거를 들어 반론한다 (이것도 적대적 루프의 일부)
- 3회째 수정 시에는 "최소한의 통과"를 목표로 스코프를 줄인다

### Victor 최종 판정

- Generator와 Discriminator가 합의하지 못할 때 Victor가 결정
- Victor는 "둘 다 틀렸다"를 선택할 수 있다
- Victor의 직감(feel)은 유효한 판정 근거

## 일일 워크플로우 예시

```
09:00  Victor: 오늘 목표 설정 (예: "보스 AI 1종")
       ↓
09:05  [G] DEV: 보스 AI 구현 시작
       ↓
10:00  [G] DEV: 1차 완성, 플레이 가능
       ↓
10:05  [D] Victor: 직접 플레이, 피드백
       "패턴이 읽히지 않는다. 텔레그래프 추가."
       "체력이 너무 높다. 2분 내 격파 가능해야."
       ↓
10:10  [G] DEV: 텔레그래프 추가, 체력 하향
       ↓
10:30  [D] Victor: 재플레이
       "텔레그래프 좋다. 체력 아직 높다."
       ↓
10:35  [G] DEV: 체력 재조정
       ↓
10:45  [D] Victor: Pass.
       ↓
10:50  커밋 + 다음 태스크
```

## 파이프라인별 담당 요약

| 파이프라인 | Generator | Discriminator | Pass 기준 |
|-----------|-----------|---------------|----------|
| GDD | Designer | PM | 수치 충돌 0, 스파이크 정렬 |
| 코드 | DEV | PM + QA | 브리프 충족, 크래시 0 |
| 리서치 | RES | Designer | 액션 아이템 1건+ |
| 비주얼 | Artist | Art Director + DEV | 톤 일치, 규격 준수 |
| 밸런스 | Designer | Simulator + QA | 체감 승인 |

---

## 위키 연동

- 각 파이프라인의 Pass/Fail 결과를 위키 daily 로그에 기록
- Fail 3회 발생 시 decisions/ 에 "보류/축소 결정" 문서 생성
- 반복 피드백 패턴이 발견되면 feedback 메모리에 저장
